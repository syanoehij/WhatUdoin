## M6-0 완료 (2026-05-10) — M6 진입 게이트 평가

- **결정**: **target-risk override 진입**(M2-0/M3-0/M4-0/M5-0과 동일 패턴). M6는 §16에서 "MCP write/edit 기능이 실제 운영 요구로 들어오면 별도 마일스톤으로 다룬다. 이 단계는 성능 개선 M1~M5의 필수 범위가 아니라 장기 운영 구조 후보"라는 조건부 후보 마일스톤이다. 정량적으로는 NO-GO 분기지만 운영 안전 boundary 정립 가치를 인정해 진입한다.
- **NO-GO 분기 근거**:
  - 현재 MCP는 read tool만 운영 — write/edit 운영 요구가 정량 측정으로 확인된 기록 없음.
  - §13 진입 게이트 조건 "MCP write/edit 운영 요구 발생 시에만 진입" 정량 미충족.
  - §16 "성능 개선 M1~M5의 필수 범위가 아니라 장기 운영 구조 후보" 명시.
- **GO override 근거**:
  1. M2/M3/M4/M5 supervisor + 분리 패턴 안정화. 추가 sidecar/IPC 비용 매우 낮음 — `WhatUdoinSupervisor`/`ServiceSpec`/`ollama_service_spec`/`media_service_spec` factory 패턴/`probe_healthz`/`STOP_ORDER` 재사용 가능.
  2. MCP write/edit owner 원칙(MCP service 직접 SQLite write 금지, 모든 write/edit는 Web API write path 경유)을 **코드 boundary로 박는** 가치. 향후 write tool 추가 시 회귀 방지. plan §9 "MCP write/edit 추가 정책 (단기 원칙)" 단언.
  3. 일정 생성/체크리스트 추가 같은 좁은 destructive 범위만 후보 — 운영 위협 낮은 단계적 도입.
  4. DB command service 도입 결정(M6-3)은 측정 기반 — 본 사이클에서는 도입 안 하고 §18 후속 후보로 유지하면 무방.
  5. 사용자 명시 진입 요청.
- **target-risk 표**:

  | 리스크 | 정량 측정 | override 사유 |
  |---|---|---|
  | MCP write/edit 운영 요구 | 미관측 | owner 원칙 코드 boundary 정립 가치 + supervisor 패턴 재사용 비용 낮음 |
  | MCP service 직접 SQLite write | 설계상 0 | M6-2에서 MCP proxy → Web API command endpoint 경유 단언 |
  | hidden-project 권한 회귀 | 41차 보안 강화로 보호 | M6-4에서 회귀 잠금 |
  | history/lock/SSE publish 정합성 | Web API write path 통합 | M6-2 boundary로 자연 보존 |
- **변경 파일**: 없음. 본 step은 결정 step.
- **다음 step 영향**: M6-1에서 write/edit tool 위험도 분류 표 작성 — destructive 분류 + 우선 후보(일정 생성/체크리스트 추가) 좁은 범위로 시작. M6-2에서 Web API command endpoint(`/api/command/*` 같은 내부 IPC) + MCP proxy 코드 boundary 도입. M6-3에서 DB command service 도입 여부는 측정으로 결정(현 사이클 미도입 가능성 높음). M6-4 종료 회귀로 hidden-project 차단/lock/history/SSE publish 통합 + MCP write owner 원칙 재확인.
