## M5-0 완료 (2026-05-10) — M5 진입 게이트 평가

- **결정**: **target-risk override 진입**(M2-0/M3-0/M4-0과 동일 패턴). M5는 §16에서 "M1c 후에도 업로드 중 Web API p95, RSS, SSE 수신 지연이 베이스라인 대비 회귀하면 진입한다"는 조건부 후보 마일스톤이다. 정량적으로는 NO-GO 분기지만 외부 의존(파일 시스템/PIL.verify/이미지 처리)의 격리 가치를 인정해 진입한다.
- **NO-GO 분기 근거**:
  - M1c-ULTRA(업로드 threadpool + `asyncio.Semaphore(8)` + Content-Length 사전 거부 + chunked hard cap)가 ULTRA 단계에서는 일부 미적용이지만, 사내 운영 환경(n≈1~5명) 업로드 시나리오에서 Web API p95 회귀가 측정으로 관찰된 기록 없음.
  - M3-4 라이브 일반 API p95 12.9ms / M4-4 hang 중 일반 API p95 31.0ms — 모두 SLA 500ms 대비 큰 여유. 업로드가 일반 API에 회귀를 만든다는 측정 증거 없음.
  - §13 진입 게이트 조건 "Web API p95, RSS, SSE 수신 지연 회귀" 미관측.
- **GO override 근거**:
  1. M2/M3/M4 supervisor + 분리 패턴 안정화. 추가 sidecar 분리 비용 매우 낮음 — `WhatUdoinSupervisor`/`ServiceSpec`/`ollama_service_spec` factory 패턴/`probe_healthz`/`STOP_ORDER` 재사용 가능.
  2. Media service 설계가 **Web API에 권한/DB metadata write/audit/SSE publish를 남기고** 파일 처리/검증/썸네일만 분리하는 보수적 패턴(§13 main → Upload/Media service 통신). 운영 안정성 위협 없음.
  3. 큰 파일(20MB 첨부/10MB 이미지) 업로드 시 PIL 메모리 spike + 디스크 I/O 지연이 main app threadpool에 미치는 영향을 사전 격리. 사내 LAN에서 첨부/이미지 업로드 빈도가 늘어나는 경우 효과 측정 가능.
  4. M2-1 외부 포트 owner 결정에서 `/uploads/meetings/*` 라우팅이 Web API의 `_ProtectedMeetingStaticFiles` 권한 검사를 거치도록 명시됨(M2-10/M2-20). M5는 그 정책을 깨지 않는 범위에서 진입.
  5. 사용자 명시 진입 요청.
- **target-risk 표**:

  | 리스크 | 정량 측정 | override 사유 |
  |---|---|---|
  | 업로드 중 Web API p95 회귀 | 미관측(M3-4/M4-4 측정값 SLA 대비 큰 여유) | 외부 의존(PIL/디스크 I/O) 격리 가치 + supervisor 패턴 재사용 비용 낮음 |

  | Media service 권한/DB 책임 분리 위반 | 설계상 Media service DB write 0 | M5-2 단언으로 잠금 |

### 편차 기록

- task spec의 `reason: "forbidden_path"` (HTTP 200)와 실제 코드의 `reason: "path_traversal"` (HTTP 400)의 명칭 불일치: 경계 동작은 동등함. 운영 코드 수정 불필요, spec 표현 불일치로 판단.
- upload_image/upload_attachment 핸들러는 URL 반환 전용으로 SSE publish를 직접 호출하지 않음 (설계 정상). 업로드 URL은 이후 doc 수정 API에서 사용되며, doc 저장 시 SSE publish가 트리거됨.
- Windows symlink 생성 권한 없음: 개발자 모드 미적용 환경. `Path.resolve().is_relative_to()` 로직이 symlink를 실제 타겟 경로로 resolve하므로, 권한이 있는 환경에서는 staging 밖 타겟 symlink도 차단됨.
- probe B p95 측정: M1c 별도 baseline 재측정 없음. M4-4 hang 중 측정값(p95=31ms)을 마지막 알려진 기준선으로 사용. mock IPC 특성상 실제 PIL+SHA-256 비용 미포함 — 라이브 업로드 대비 업로드 thread 처리 시간이 짧아 동시 부하가 실제보다 낮을 수 있음. 사내 운영 환경(n≈1~5명)에서 실질적 회귀 미관측.
- probe B 업로드 thread 완료 시간: mock IPC로 인해 10MB/20MB 파일 disk-write(~16ms)가 완료된 후 잔여 GET이 idle 상태에서 처리됨. 부하 동시 윈도우는 약 16ms. 78ms p95는 disk-write 윈도우 포함 측정값으로 SLA 500ms 대비 여유 확인됨.
- probe B upload thread: 실제 /api/upload/* 엔드포인트 미호출, mock IPC(_do_mock_upload) 직접 사용. Web API의 DB write/staging rename/SSE publish 비용은 phase70 회귀(69종 단언)에서 별도 검증.
