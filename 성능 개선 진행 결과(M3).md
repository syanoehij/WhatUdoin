## M3-0 완료 (2026-05-10) — M3 진입 게이트 평가

- **결정**: **target-risk override 진입**. 정량적으로는 NO-GO 분기지만 운영 안전판 가치를 인정해 진입한다(M2-0과 동일 패턴).
- **NO-GO 분기 근거**:
  - M1a-7 baseline 50 VU 측정에서 `database is locked` 0건. scheduler write가 일반 API p95에 측정 가능한 회귀를 만든 적 없음.
  - multi-worker 전환은 §13 "거부 — main app multi-worker 전환"에서 명시 거부. uvicorn 1 worker 유지.
  - §13 진입 게이트 두 조건(scheduler write 회귀 측정 / multi-worker 검토) 모두 미충족.
- **GO override 근거**:
  1. M2 supervisor 골격이 이미 도입돼(M2-9/17 적용) 추가 sidecar 분리 비용이 낮다. `WhatUdoinSupervisor`/`ServiceSpec`/`crash_history`/`probe_healthz` 재사용 가능.
  2. §11 startup maintenance 단일 owner 정책이 코드로 명시되지 않은 상태. 현재 `app.py` lifespan이 `finalize_expired_done()` 즉시 실행 + APScheduler 시작을 같이 하므로, 향후 회귀 방지를 위해 owner 표를 코드화하는 가치가 있다.
  3. M3는 회사 반입 운영 안정화 측면에서 야간 정리 작업이 main app 응답성에 영향을 주지 않게 격리하는 의미. 사내 운영 환경에서 이미 1~5 VU 정도라 정량 회귀가 안 보일 뿐, 야간에 main app 한 프로세스가 정리 + 알림 + Web API 모두 처리하는 구조는 장애 발생 시 영향 범위가 크다.
  4. 사용자 명시 진입 요청.
- **target-risk 표**:

  | 리스크 | 정량 측정 | override 사유 |
  |---|---|---|
  | scheduler write로 일반 API p95 회귀 | 미관측(M1a-7) | 사내 1~5 VU 운영에서 정량 미관측이라 일반화 불가, 야간 정리 + Web API 동거 구조의 장애 영향 격리 가치 |
  | multi-worker 전환 검토 | §13 거부 | 운영 진입과 별개로 startup maintenance owner 정책 코드화 자체에 가치 |
- **변경 파일**: 없음. 본 step은 결정 step.
- **다음 step 영향**: M3-1에서 §11 표(6개 작업의 owner) + 코드 위치(현재 lifespan 즉시 실행 vs APScheduler 트리거) 매핑이 첫 작업이다. M3-2에서 APScheduler를 별도 프로세스로 분리하기 전에 owner 표를 먼저 코드 주석/문서로 박아 놓는다.
