# 팀 기능 구현 계획

## 1. 운영 전제

- 현재 실사용자는 약 6명이다.
- 향후 50명 이상까지 늘어날 수 있다는 전제로 설계한다.
- 유지보수는 1명이 맡고 있으며, 전담 운영 인력이 없다.
- 별도 DB 서버, Redis, nginx, 별도 웹서버처럼 운영 부담이 큰 외부 인프라는 도입하지 않는다.
- 기본 운영 구조는 `FastAPI + SQLite + 단일 PC + 단일 EXE/트레이`를 유지한다.
- 목표는 최고 확장성이 아니라, 혼자 운영 가능한 범위에서 50명 규모까지 안정적으로 버티는 것이다.

## 2. 역할 체계

**시스템 관리자(admin)**
- 팀 생성/삭제
- 팀 관리자 지정
- 모든 팀의 멤버 관리 (수락·거절·추방·IP 설정)
- 히든 프로젝트 포함 전체 데이터 관리
- 웹앱 슈퍼유저로 동작한다. `/admin` 관리 화면뿐 아니라 일반 업무 화면(`/`, `/doc`, `/check`, `/calendar`, `/kanban`, `/gantt`, `/project-manage`)에도 진입 가능하며, 모든 팀의 자료를 조회·작성·수정·삭제·복원할 수 있다.
- `/admin`은 팀/계정/IP/가입 승인/강제 이관처럼 운영 전용 기능에 집중한다. 문서·체크·일정·프로젝트의 일반 편집은 기존 업무 화면을 슈퍼유저 권한으로 사용하는 방향을 우선한다. 이렇게 해야 관리 콘솔이 과도하게 비대해지지 않는다.
- admin은 실제 팀 멤버가 아니므로 일반 멤버 목록이나 담당자 후보에는 노출하지 않는다. 다만 일반 업무 화면에서 새 자료를 만들거나 팀 자료를 필터링할 때는 프로필 메뉴의 **현재 작업 팀**을 선택해 그 팀 컨텍스트로 작업한다.
- 현재 작업 팀이 필요한 쓰기 요청에서 admin이 팀을 선택하지 않았거나 존재하지 않는 팀을 보낸 경우 서버에서 400/403으로 거부한다.
- MCP 토큰 발급은 웹앱 슈퍼유저 권한과 별도 보안 표면으로 본다. 1차 구현에서는 admin MCP 토큰 발급 차단을 유지한다(섹션 16-1 참조).
- `users.role = admin`으로 식별
- 특정 팀에 소속되지 않는다.
- `user_teams`에 admin row를 만들지 않는다.
- DB 상 `users.team_id`도 NULL로 둔다. 신규 admin 시드는 어떤 팀에도 자동 배정하지 않는다.
- 단, 권한 판단에서는 모든 팀에 접근 가능한 가상 전역 범위를 부여한다.
- 팀 목록, 멤버 목록, 담당자 목록, 히든 프로젝트 멤버 후보, 일반 사용자 자동완성, MCP 일반 사용자 조회에 노출되지 않는다.

**팀 관리자**
- 시스템 관리자가 팀 생성/관리 페이지에서 소속 멤버 중 지정
- **팀별로 관리자 여부가 달라진다.** A팀 관리자이면서 B팀 일반 멤버일 수 있다.
- `user_teams.team_role = admin`으로 팀별로 식별 (`users.role`에 별도 값 없음)
- 일반 멤버로서 팀의 모든 기능(캘린더·문서·체크·칸반·간트)을 그대로 사용
- 추가로 소속·관리자 권한이 있는 팀에 대해 아래 관리 기능을 사용할 수 있다.
  - 멤버 수락·거절·추방
  - 메뉴 외부 노출 여부 설정
- 팀 관리자는 IP 자동 로그인 관련 권한을 어떤 형태로도 갖지 않는다. 타인/임의 IP whitelist 등록·해제·대상 사용자 지정은 시스템 관리자(admin) 전용이며, 팀 관리자 UI에서는 IP 정보 자체를 노출하지 않는다(섹션 10 참조). 단, 일반 사용자의 본인 PC 자동 로그인 등록·해제는 설정 화면에서 허용한다.
- 히든 프로젝트: 멤버에 포함된 경우에만 조회, `projects.owner_id`인 경우에만 편집·멤버 관리

**멤버**
- `users.role = member`
- 여러 팀에 소속 가능
- 소속 팀의 기능 이용

> 코드 영향: 현재 `auth.is_editor`(`role IN ('editor','admin')`)와 `_require_editor` 등 30+ 군데 호출부는 단순 치환하면 안 된다. `users.role`은 `admin / member`로 전환하되, 일반 멤버 여부(`is_member`: member만)와 슈퍼유저 여부(`is_admin`: admin만)를 분리한다. 일반 사용자용 쓰기 API는 `member 또는 admin`을 허용하되, admin일 때는 현재 작업 팀을 명시적으로 검증해야 한다. DB의 `role` 값은 `editor` → `member`로 마이그레이션한다(섹션 13).

## 3-1. 현재 DB 상태와 수용 방향

실제 `whatudoin.db`와 `database.py`를 확인한 결과, 현재 DB는 이미 단일 DB 안에 팀 관련 컬럼을 일부 갖고 있지만 팀 기능 전체를 담기에는 식별 기준이 섞여 있다.

**현재 구조의 핵심 제약**
- `users.team_id`는 단일 소속만 표현한다.
- `projects.name`은 전역 `UNIQUE`다.
- `events.project`와 `checklists.project`는 프로젝트 이름 문자열로 연결된다.
- `project_milestones`와 `project_members`는 이미 `project_id`로 연결된다.
- `meetings`, `events`, `checklists`, `projects`에는 `team_id`가 있지만 일부 레거시 데이터는 `team_id NULL`일 수 있다.
- `teams`에는 `deleted_at`이 없고, 현재 삭제 API는 hard delete 방식이다.
- 업로드 파일은 현재 `/uploads/meetings/YYYY/MM/...` 전역 경로를 사용한다.
- `users.password`, `pending_users.password`는 평문 컬럼이다.
- `user_ips`는 사용자 전역 IP 목록이며 팀별 IP가 아니다.
- `init_db`는 신규 환경에서 `teams`에 "관리팀"을 자동 시드하고 초기 admin 계정을 그 팀(`team_id=1`)으로 묶어 만든다. "관리팀"은 한글이라 섹션 4 정규식 `^[A-Za-z0-9_]+$`에도 위반된다.
- `notifications`에는 `team_id` 컬럼이 없어 다중 팀 환경에서 알림 컨텍스트를 구분할 수 없다.
- `users.role` 값은 현재 `admin / editor` 두 가지이며 코드 전반(`auth.is_editor`, `_require_editor`, `can_edit_*`)이 이 키워드에 묶여 있다.

**DB 개선 원칙**
- 신규 기능의 표준 식별자는 `team_id`, `user_id`, `project_id`다.
- 프로젝트명은 화면 표시와 사용자 입력용이며, 서버 내부 권한 판단·수정·삭제·내보내기·MCP 필터의 기준으로 단독 사용하지 않는다.
- 기존 `events.project`, `checklists.project` 문자열 컬럼은 1차 마이그레이션 동안 읽기 호환용으로 유지하되, 신규 쓰기는 `project_id`를 우선 저장한다.
- 신규 데이터의 `team_id`는 원칙적으로 `NOT NULL`이다. **레거시 `team_id NULL` 데이터는 작성자/프로젝트 기준으로 팀을 추론해 배정하며, 추론 실패 시 NULL을 유지하고 `team_migration_warnings`에 기록한다(섹션 13 확정안).** 임의 "기본 팀"으로 일괄 배정하지 않는다(섞이면 분리 비용이 큼). NULL 잔존 row는 보수적 가시성(작성자/owner 본인 + admin 슈퍼유저)으로 처리한다.
- **예외**: `meetings.is_team_doc=0`(개인 문서) 중 작성자가 팀 미배정 상태에서 작성한 신규 row는 `team_id = NULL`을 허용한다. 작성자 본인 한정 가시성으로 처리한다(섹션 7·8).
- SQLite에서 기존 `UNIQUE` 제약 제거가 까다로우므로, `projects`처럼 전역 UNIQUE를 제거해야 하는 테이블은 안전한 테이블 재생성 마이그레이션을 별도 단계로 수행한다.
- 테이블 재생성이 필요한 마이그레이션은 반드시 DB 백업 후 수행하고, 실패 시 기존 DB를 그대로 복구할 수 있어야 한다.
- `settings`에 migration marker를 남겨 같은 마이그레이션이 반복 실행되지 않게 한다.

**권장 목표 제약**
- `users.name_norm`은 `is_active`와 무관하게 전체 사용자에서 유일하다. 비활성 사용자도 계정명을 계속 점유한다.
- `teams.name_norm`은 전체 팀에서 유일하다. 삭제 유예기간 중인 팀도 이름을 점유한다.
- 팀이 90일 후 완전 삭제되면 같은 팀 이름을 다시 사용할 수 있다.
- `projects`는 같은 팀 안에서 `name_norm`이 유일하다. 다른 팀은 같은 `name_norm`을 사용할 수 있다.
- 삭제 또는 휴지통 상태의 프로젝트도 완전 삭제 전까지는 같은 팀 안에서 이름을 점유한다. 복원 충돌을 막기 위함이다.
- `user_teams`는 `(user_id, team_id)`를 유일하게 유지한다. 재신청은 row 추가가 아니라 기존 row의 `join_status` 갱신으로 처리한다.
- `user_ips`는 `type = 'whitelist'`인 `ip_address`가 전역에서 유일해야 한다.
- `team_menu_settings`는 `(team_id, menu_key)`를 유일하게 유지한다.

**대표 팀 결정 규칙**
- 대표 팀 후보는 `user_teams.join_status = approved`이고 `teams.deleted_at IS NULL`인 팀만 포함한다.
- 현재 작업 팀 후보도 대표 팀 후보와 동일하게 삭제 예정 팀을 제외한다.
- 소속 팀이 1개이면 그 팀이 대표 팀이다.
- 소속 팀이 2개 이상이면 가장 먼저 수락된 팀(`joined_at` 기준)이 대표 팀이다.
- 삭제 예정 팀만 남은 사용자는 팀 미배정 사용자와 동일하게 처리한다.
- 별도 대표 팀 선택 UI는 제공하지 않는다.

## 4. 팀 생성 및 URL 정책

- 시스템 관리자만 팀을 생성할 수 있다.
- 팀 생성 시 팀 이름이 외부 공개 주소의 URL 세그먼트가 된다.
  - 예: 팀 이름이 `ABC`이면 `http://localhost:8000/ABC`

**팀 이름 규칙**
- 영문 대문자/소문자, 숫자, 언더바(`_`)만 허용 — 정규식: `^[A-Za-z0-9_]+$`
- 대소문자 중복 방지 (case-insensitive 유일성 검사): `ABC`와 `abc`는 중복으로 처리
- URL은 생성 시 입력한 대소문자 그대로 사용한다. `ABC`로 생성하면 `/ABC`만 유효하며 `/abc`는 404.
- 팀 이름은 생성 후 변경할 수 없다. 신중하게 정한다.

**예약 경로 (팀 이름으로 사용 금지)**
- `api`, `admin`, `doc`, `check`, `kanban`, `gantt`, `calendar`, `mcp`, `mcp-codex`, `uploads`, `static`, `settings`, `changelog`
- `register`, `project-manage`, `ai-import`, `alarm-setup`, `notice`, `trash`, `remote`, `avr`, `favicon.ico`
- FastAPI 기본 문서 경로: `docs`, `redoc`, `openapi.json`
- 구현 시에는 하드코딩 목록과 실제 등록 route 목록을 함께 검사해 누락을 막는다.

## 5. 팀 삭제 및 복원

**팀 삭제 권한은 시스템 관리자(admin) 전용이다.** 팀 관리자(`user_teams.team_role = admin`)는 팀을 삭제할 수 없으며, `/admin` 화면에서 시스템 관리자만 삭제·복원 작업을 수행한다. 즉시 삭제되지 않고 90일의 유예기간을 거친다.

**삭제 요청 시**
- 해당 팀에 대한 모든 접근이 즉시 차단된다 (문서·체크·캘린더 등 포함).
- `teams.deleted_at`에 삭제 예약 시각을 기록한다 (soft delete).
- 유예기간 중 `/팀이름`으로 접속하면 "이 팀은 삭제 예정입니다" 안내 페이지를 표시한다. 안내 페이지에는 팀 메시지만 표시하며, **"계정 가입"·"팀 신청"·공개 포털 데이터·계정 가입 버튼은 노출하지 않는다.** 가입을 원하는 사용자는 `/`로 직접 이동해 가입한다.
- 유예기간 중인 팀은 신규 팀 신청 후보에서 제외된다. `/`의 팀 목록·드롭다운, `/팀이름` 공개 포털의 팀 신청 버튼 모두 차단된다.

**유예기간 중 복원**
- 시스템 관리자는 팀을 복원할 수 있다.
- 복원 즉시 해당 팀의 모든 데이터에 다시 접근할 수 있다.
- 유예기간 중 `user_teams` rows는 삭제하지 않고 그대로 유지한다. 접근 차단은 서버에서 `teams.deleted_at IS NOT NULL`로 처리하며, 복원 시 별도 작업 없이 멤버 관계가 자동 복구된다.
- 유예기간 중인 팀은 대표 팀/현재 작업 팀 후보에서 제외한다.
- 사용자에게 삭제 예정 팀만 남아 있으면 `/`에서 팀 미배정 사용자처럼 팀 목록과 팀 신청/대기 상태를 보여준다.

**90일 후 자동 완전 삭제**
- 매일 새벽 DB 백업 완료 이후 삭제 스케줄러가 실행된다.
- **삭제 순서 — 명시적 12단계** (TODO #23이 source of truth. 신규 팀 종속 테이블 추가 시 이 목록도 함께 갱신):

## 6. 로그인 / 인증

**로그인 방식 — 두 가지가 공존한다**
- IP 기반 자동 로그인: 관리자가 등록한 IP에서 접속하면 자동 로그인
- 아이디+비밀번호 직접 입력: 미등록 IP에서 접속 시 사용
  - 아이디: 영문(대소문자) + 한글 + 숫자만 허용. 밑줄·공백·특수문자 차단.
  - 아이디 중복 판단은 대소문자 무관(case-insensitive)으로 처리한다. `Kim`과 `kim`은 같은 계정이다. 저장/표시는 최초 입력 대소문자를 보존하되 로그인·중복 검사는 정규화 이름(`name_norm`)으로 수행한다.
  - 비밀번호: 영문과 숫자를 모두 포함해야 한다.
- 일반 사용자 직접 로그인은 기존 비밀번호 단독 로그인을 폐기하고 이름(아이디)+비밀번호 입력으로 전환한다.
- 시스템 관리자(admin)는 현재처럼 별도 `/api/admin/login` 경로와 짧은 세션을 유지한다. admin은 IP 자동 로그인 대상이 아니며 일반 사용자 로그인 화면/자동 로그인 흐름에 섞지 않는다.
- **일반 `/api/login`은 admin 자격증명을 인지하지 않는다.** 서버는 일반 로그인 조회 시 `users.role = admin`을 제외한 사용자만 매칭한다. admin 이름으로 일반 로그인을 시도하면 "이름 또는 비밀번호가 올바르지 않습니다"로 응답하며, admin 계정의 존재 여부도 노출하지 않는다(이름 노출 최소화). admin은 반드시 `/api/admin/login` 경로를 사용해야 한다.

**비밀번호 정책 적용 시점**
- **계정 가입**: 새로 입력하는 비밀번호는 영문+숫자 동시 포함 여부를 서버에서 검증해 미충족 시 계정 생성을 차단한다.
- **일반 비밀번호 변경**: 동일 정책을 검증한다.
- **마이그레이션 시 평문 → hash 일괄 변환**: 이 단계에서는 정책을 검증하지 않는다. 기존 평문이 정책을 위반하더라도 변환 자체는 통과시키고, 위반 비밀번호는 사용자가 자발적으로 변경할 때 정책에 맞춰 갱신된다.

**비밀번호 보안**
- 이름(아이디)은 평문으로 저장한다.
- 비밀번호는 복호화 불가능한 salted hash로 저장한다.
- 평문 비밀번호는 마이그레이션 시점에 일괄 hash로 변환한다. 변환 후에는 사용자가 기존에 쓰던 평문 비밀번호로 그대로 로그인할 수 있다(hash 매칭이 정상 동작).
- 강제 변경 절차는 두지 않는다. 마이그레이션 후 정책 위반 비밀번호가 일부 계정에 남을 수 있으나, 운영 인력이 직접 안내해 자발적으로 변경하도록 한다. 운영 규모(<10명)를 전제로 한 결정이며, 향후 규모가 커지면 별도 마이그레이션으로 강제 변경 메커니즘을 추가할 수 있다.
- **admin도 동일하게 적용한다**. 시스템 관리자(admin) 계정의 평문 비밀번호도 정책 검증 없이 hash로 변환되며, 변환 후 기존 평문 비밀번호로 그대로 로그인한다. admin은 IP 자동 로그인 대상이 아니므로(아래 항) 항상 아이디+비밀번호 입력 경로를 사용하지만, 일반 사용자와 마찬가지로 강제 변경·외부 IP 차단·1회 예외 처리는 없다.

**IP 자동 로그인 운영 전제**
- 회사 운영상 1인당 고정 IP 1개가 제공된다는 전제를 둔다.
- whitelist IP는 한 명의 사용자에게만 등록할 수 있다.
- 같은 IP를 여러 사용자에게 whitelist로 중복 등록하려 하면 서버에서 차단한다.
- 시스템 관리자(admin)는 IP 자동 로그인 대상에서 제외한다.
- **기존 운영 DB에 admin의 `user_ips` whitelist row가 있으면 마이그레이션(Phase 2)에서 `type='history'`로 강등한다(접속 이력 정보는 보존, 자동 로그인 효력만 제거). row 삭제는 하지 않으며, 향후 admin 관리·진단용 정보로 유지된다.**

**자동 로그인 대상 IP 등록 경로 — 두 가지가 공존한다**

1. **사용자 자체 등록** (설정 화면)
   - 로그인한 일반 사용자는 설정 화면에서 "현재 PC를 자동 로그인 대상으로 등록" 토글을 켜면, 그 요청의 클라이언트 IP가 본인 `user_ips`에 `type = 'whitelist'`로 등록된다.
   - **토글 ON 시 확인 모달을 띄운다.** 본문 예시: *"이 PC의 IP를 자동 로그인 대상으로 등록합니다. 등록 후에는 이 PC에서 로그인 없이 본인 계정으로 자동 진입됩니다. 공용 PC·공유 와이파이·외부 네트워크에서는 보안상 위험하니 본인 전용 PC에서만 사용하세요. 등록 후 다른 PC로 이동했다면 설정에서 즉시 해제하세요."* 모달에서 "확인"을 눌러야 실제로 whitelist 등록이 완료된다.
   - 같은 화면에서 본인 whitelist를 직접 해제할 수도 있다(토글 off → `type = 'history'`로 강등 또는 row 삭제). 해제 시에는 별도 확인 모달 없이 즉시 처리한다.
   - 등록 시도 시 그 IP가 이미 다른 사용자에게 whitelist로 등록돼 있으면 서버가 409로 거부하고 "이 PC는 다른 사용자의 자동 로그인 대상으로 등록되어 있습니다. 시스템 관리자에게 문의하세요." 안내를 표시한다.
   - admin 세션에서는 이 토글을 노출하지 않는다(admin은 IP 자동 로그인 대상이 아님).

2. **시스템 관리자 등록** (`/admin` 화면)
   - 시스템 관리자는 임의의 사용자에게 임의의 IP 문자열을 직접 입력해 whitelist로 등록할 수 있다.
   - 사용자가 한 번도 접속한 적 없는 새 IP도 등록 가능(예: 운영 정책상 미리 부여).
   - 충돌 시(다른 사용자에게 이미 whitelist) 서버가 409로 거부한다. admin이 충돌 IP를 다른 사용자에게 옮기려면 먼저 기존 whitelist를 해제·삭제한 뒤 새로 등록한다.

**계정 가입과 팀 신청 — 두 단계로 분리한다**

1단계: 계정 가입
- 로그인 없이 계정을 만들 수 있다.
- 계정 가입 화면에서는 이름(아이디), 비밀번호만 입력한다. 팀 선택은 불가능하다.
- 이름(아이디) 중복 여부를 신청 시점에 대소문자 무관으로 확인한다. `users.name_norm`에 이미 존재하면 계정 생성을 차단한다. `Kim`과 `kim`은 같은 계정으로 취급한다.
- 비밀번호 중복 여부는 더 이상 검사하지 않는다. salted hash 구조에서는 같은 비밀번호 여부를 가입 차단 조건으로 쓰지 않는다.
- **예약 사용자명 차단**: `admin`, `system`, `root`, `guest`, `anonymous`는 가입 시 사용 불가. 대소문자 무관. 시각적 혼동 방지와 운영 보안을 위한 사전 차단.
- 계정이 생성되면 `users.role = member`, `users.team_id = NULL`, `is_active = 1`로 저장한다. `user_teams` row는 만들지 않는다.
- **가입 완료 직후**: 서버는 자동으로 세션을 생성하고 사용자를 `/`로 리다이렉트한다. 사용자는 별도 로그인 절차 없이 바로 팀 목록 + 팀 신청 버튼 + "내 자료" 영역이 있는 화면을 보게 된다.
- 팀에 수락되기 전까지 내부 업무 앱(캘린더·문서·체크·칸반·간트)에는 진입할 수 없다.
- 신규 표준 경로에서는 `pending_users`를 사용하지 않는다. **기존 `pending_users` row는 마이그레이션 시 status 무관하게 모두 자동 삭제된다(섹션 13 결정)**. 신청 대기 중이던 사용자는 운영자가 별도로 안내해 새 가입 흐름(`/`의 "계정 가입" 버튼)으로 다시 신청한다. 자동 이관·운영자 수동 이관 옵션은 제공하지 않는다(코드·UI 단순성 우선).

2단계: 팀 신청
- 대상: 계정이 있는 모든 사람. 현재 소속 팀이 없어도 포함.
- 진입 경로:
  - 팀이 없는 경우: `/팀이름` 공개 포털의 팀 신청 버튼 (이름·비밀번호 입력 없이 팀만 선택)
  - 팀이 있는 경우: `/팀이름` 공개 포털의 팀 신청 버튼 또는 설정 페이지의 '팀 추가 신청' 메뉴
- 신청할 팀을 1개 선택한다.
- `user_teams`에 `join_status = pending` row가 1개라도 있으면 추가 신청 불가.
- 시스템 관리자 또는 해당 팀의 팀 관리자가 수락 또는 거절한다.
- 수락되면 해당 팀이 소속 팀 목록에 추가된다. 기존 팀 소속은 유지된다.
- 거절 또는 추방 후 재신청 가능하다.
- 구현 관점: 팀 신청은 항상 `user_teams`에 `join_status = pending` row 삽입으로 처리한다. 수락 시 `approved`로 업데이트, 거절/추방 시 `join_status = rejected`로 유지한다(이력 보존). 재신청은 같은 `(user_id, team_id)` row를 `pending`으로 갱신한다.

## 7. 비로그인 / 로그인 URL 동작

**비로그인 사용자**
- `/` 접속 시: 생성된 팀 목록 + 로그인 버튼 + 계정 가입 버튼 표시
- 팀을 클릭하면 `/팀이름`으로 이동하며 해당 팀의 공개 설정된 데이터를 읽기 전용으로 표시. 그 화면에도 "계정 가입" 버튼을 노출해 비로그인 사용자가 곧바로 가입할 수 있게 한다(가입 후 자동 로그인 + `/`로 이동 — 섹션 6 1단계).

**팀에 배정되지 않은 로그인 사용자**
- 비로그인 유저처럼 `/`에서 팀 목록이 노출된다.
- 로그인 상태이므로 각 팀 옆에 팀 신청 버튼이 표시된다.
  - 신청 전: "팀 신청" 버튼
  - 신청 후 대기 중: "가입 대기 중" 버튼 (비활성)
  - 거절 또는 추방 후: 다시 "팀 신청" 버튼 표시 (재신청 가능)
- 팀에 수락되어야 내부 업무 앱(캘린더·문서·체크·칸반·간트)에 진입할 수 있다.
- **다만 본인 작성 자료의 통합 노출을 위해 별도 "내 자료" 영역을 제공한다.**
  - `/` 화면 하단에 "내 자료" 섹션을 두고 본인이 작성한 개인 문서(`is_team_doc=0`)만 표시한다. 일정·체크·팀 문서는 표시하지 않는다(이들은 모두 팀 컨텍스트가 필요).
  - **알림은 표시하지 않는다.** 팀 미배정 상태에서는 어떤 팀의 알림도 의미를 가지지 않으므로 알림 카드·뱃지·페이지 자체를 노출하지 않는다. 팀에 가입·수락된 시점부터 알림 영역이 다시 활성화되며, 그 시점부터 발생하는 알림만 받게 된다(과거 미수신 알림을 소급 노출하지 않는다).
  - 이 영역은 팀 미배정 사용자 + 팀이 모두 삭제 예정인 사용자에게만 노출된다. 팀 배정 사용자는 현재 작업 팀 컨텍스트로 동일 데이터를 본다.
  - 본인 외에는 이 영역의 데이터에 접근할 수 없다. `team_share=1` 공유 문서는 이 화면에 표시되지만 다른 사용자에게는 현재 작업 팀 일치 조건이 그대로 적용된다(섹션 8).
  - **신규 개인 문서 작성 허용**: 팀 미배정 사용자도 "내 자료" 영역에서 새 개인 문서를 작성할 수 있다. 이때 `team_id = NULL`로 저장한다(섹션 3 규칙). `team_share`는 적용 의미가 없으므로 UI에서 비활성화한다. 가시성은 작성자 본인 한정.

**팀에 배정된 로그인 사용자**
- `/` 접속 시: 내부 업무 앱. 프로필 이름 메뉴의 "팀 변경"에서 현재 작업 팀을 선택한다.
- `/팀이름` 접속 시: 해당 팀 공개 포털 표시. 홈 버튼을 누르면 `/`로 이동.

**시스템 관리자 로그인 사용자**
- `/admin`뿐 아니라 일반 업무 화면에도 슈퍼유저로 진입할 수 있다.
- `/` 접속 시: 내부 업무 앱 표시. 프로필 이름 메뉴의 "팀 변경"에서 현재 작업 팀을 선택한다.
- `/팀이름` 접속 시: 일반 사용자와 동일하게 해당 팀 공개 포털을 볼 수 있다.
- `/admin` 내부에서는 모든 팀에 대한 전역 관리 권한을 가진다. 단, 이 권한은 실제 `user_teams` 소속 row가 아니라 `users.role = admin` 기반의 가상 전역 범위다.

**현재 작업 팀 선택**
- 이름/프로필 메뉴에 "팀 변경" 버튼을 둔다. 사용자는 여기서 현재 작업 팀을 선택한다.
- 팀을 하나 선택하면 캘린더·문서·체크·칸반·간트·프로젝트 관리 화면이 모두 그 팀 기준으로 동작한다.
- 기존 칸반·간트·캘린더 화면의 별도 팀 선택 드롭다운은 제거한다. 팀 컨텍스트가 여러 곳에 흩어지지 않게 하기 위함이다.
- 프로필 영역에는 현재 작업 팀 이름을 함께 표시한다. 예: `홍길동 · HW팀`, admin은 `admin · HW팀(슈퍼유저)`처럼 표시한다.
- 저장 매체는 `work_team_id` 쿠키를 1차 초기값으로 사용한다. SSR 첫 페이지 렌더 시점에 서버가 현재 작업 팀을 알아야 하기 때문이다. localStorage는 백업/UX 보조 용도다.
- 클라이언트가 "팀 변경"에서 팀을 바꾸면 `/api/me/work-team` 같은 전용 API로 서버 검증을 거쳐 쿠키를 갱신하고, 현재 페이지를 새 팀 컨텍스트로 다시 로드한다.
- 처음 로그인 시 쿠키가 없으면 서버는 대표 팀(joined_at 기준 가장 먼저 수락된 팀)을 현재 작업 팀으로 제안하고 쿠키에 기록한다. admin은 모든 비삭제 팀 중 선택 가능하며, 기본값은 첫 번째 팀 또는 마지막 선택 팀이다.
- 저장된 작업 팀이 삭제 예정 상태가 되었거나 사용자의 approved 소속에서 빠진 경우에는 서버가 쿠키 값을 무시하고 새 대표 팀을 다시 계산해 쿠키를 갱신한다.
- API 요청 시 프론트엔드는 명시적으로 `team_id` 파라미터를 함께 보낸다. 서버는 그 값을 `require_work_team_access`로 검증한다. 쿠키는 기본 선택값일 뿐, 권한 판단은 항상 서버 검증을 통과한 `team_id`로 한다.

정리:

```text
/         비로그인                      → 팀 목록 + 로그인/계정 가입 버튼
/         로그인, 팀 미배정             → 팀 목록 + 팀 신청/대기중 버튼 + "내 자료" 영역
/         로그인, 팀 배정 완료          → 내부 업무 앱 (프로필 메뉴에서 현재 작업 팀 선택)
/         로그인, 시스템 관리자         → 내부 업무 앱 슈퍼유저 진입 가능 + /admin 직접 진입 가능
/팀이름   비로그인                      → 팀 공개 포털 (공개 설정된 항목만 읽기 전용) + 계정 가입 버튼
/팀이름   로그인, 팀 미배정             → 팀 공개 포털 표시 (이미 로그인 상태이므로 팀 신청 버튼은 팀만 선택하면 바로 신청 가능, 이름·비밀번호 입력 없음)
/팀이름   로그인, 팀 배정 완료          → 해당 팀 공개 포털 표시 (홈 버튼으로 / 이동)
/팀이름   로그인, 시스템 관리자         → 해당 팀 공개 포털 표시
/팀이름   유예기간 중 삭제된 팀, 비-admin → 팀 삭제 예정 안내 페이지
/팀이름   유예기간 중 삭제된 팀, 시스템 관리자 → 팀 삭제 예정 안내 + /admin 관리 링크
```

**`/팀이름` 공개 포털에서의 계정/팀 신청 버튼 상태**

| 접근 상태 | 버튼 표시 |
|-----------|----------|
| 비로그인 | "계정 가입" → 이름·비밀번호만 입력. 가입 완료 후 로그인 상태에서 해당 팀 신청 가능 |
| 로그인, 해당 팀 미소속 | "팀 신청" → 해당 팀이 미리 선택된 상태로 바로 신청 |
| 로그인, 해당 팀 pending 대기 중 | "가입 대기 중" (비활성) |
| 로그인, 해당 팀 approved 소속 | 버튼 없음 |
| 로그인, 거절 또는 추방 후 | "팀 신청" 재표시 |

중요 원칙:
- URL은 권한 경계가 아니다.
- `/팀이름`은 public portal context일 뿐이다.
- 비공개 데이터 접근은 항상 서버에서 사용자 소속팀과 visibility를 검증한다.

## 8. 화면별 팀 기준

대부분의 내부 업무 화면은 현재 선택한 **작업 팀**을 기준으로 동작한다.

팀별로 분리되는 기능:
- 캘린더 (기존)
- 칸반 (기존)
- 간트 (기존)
- 문서 (신규)
- 체크 (신규)

예외:
- `내 스케줄` 계열 화면: 소속된 모든 팀의 정보를 통합해서 표시
- `/admin` 화면: 전체 팀 데이터 조회 가능
- 알림(`notifications`): 현재 작업 팀과 무관하게 항상 **현재 승인·비삭제 소속 팀** 알림만 통합 표시. 결과 항목에 발생 팀 이름을 함께 표기해 출처를 구분한다. (분류상 `내 스케줄` 계열에 해당)
  - **추방·탈퇴 처리**: 사용자가 더 이상 소속되지 않은 팀의 과거 알림은 알림 목록에서 숨긴다. row 자체는 삭제하지 않고 화면 필터로만 제외(`user_teams.join_status = approved` AND `teams.deleted_at IS NULL`인 팀의 알림만 노출). 다시 그 팀에 가입·수락되면 과거 알림도 재노출된다.
  - **알림 클릭 동작**: 알림이 다른 팀에서 발생한 경우, 클릭 시 현재 작업 팀을 그 팀(`notifications.team_id`)으로 자동 전환한 뒤 해당 항목 상세 화면으로 이동한다. 전환 직후 헤더에 "OO팀으로 전환됨" toast를 표시해 사용자가 컨텍스트 변경을 인지하도록 한다. `team_id`가 NULL("팀 미상")인 알림은 작업 팀을 전환하지 않고 알림 페이지에서만 보여준다.
- 팀 미배정 사용자 전용 "내 자료" 영역: 본인 작성 개인 문서(`is_team_doc=0`)만 표시. 알림은 노출하지 않는다 (섹션 7)

**개인 문서(`is_team_doc=0`) 가시성**
- **작성자 본인**: 현재 작업 팀·소속 팀과 무관하게 본인의 모든 개인 문서가 항상 통합 노출된다. 본인이 작성한 자료이므로 모든 팀에서 추방돼 소속 팀이 0개여도 "내 문서" 목록에서 계속 볼 수 있다.
- **`team_share=1` 개인 문서를 다른 팀 멤버가 볼 때**: 그 문서의 `team_id` 팀 멤버에게 현재 작업 팀이 그 팀일 때만 읽기 노출된다. 작업 팀이 다른 팀이면 같은 팀 멤버에게도 보이지 않는다.
- **`team_id IS NULL`인 개인 문서**: 팀 미배정 사용자가 작성한 문서. 작성자 본인 외에는 어떤 사용자도 접근할 수 없다(`team_share` 값과 무관). 다른 팀 멤버에게 노출하려면 작성자가 팀에 배정된 후 팀 문서로 새로 작성하거나 본문을 복사해야 한다.
- 작성자가 해당 팀에서 추방·탈퇴해도 문서의 `team_id`는 변하지 않으며, 작성자 본인은 위 첫 줄에 따라 계속 접근 가능하다. 다른 팀 멤버는 작업 팀 일치 조건을 충족하면 `team_share=1`인 한 계속 읽을 수 있다.

**체크(`checklists`) 가시성**
- 체크는 프로젝트와 묶인 팀 자료다. 현재 작업 팀 컨텍스트에 의존하며, 작성자가 그 팀에서 추방·탈퇴하면 본인에게도 보이지 않는다. 개인 문서와 다르게 통합 노출하지 않는다.

운영 원칙:
- 로그인 사용자는 프로필 이름 메뉴의 "팀 변경"으로 현재 작업 팀을 선택해 작업한다.
- 내부 작업 화면에서는 팀 URL을 사용하지 않는다.
- 서버는 매 요청마다 사용자가 해당 팀 소속인지, 또는 admin 슈퍼유저 범위인지 검증한다.

**현재 구현 상태 (팀 기능 도입 전):**
- 캘린더: `_filter_events_by_visibility`에서 `users.team_id` 단일 비교로 팀 경계 구현됨. `user_teams` 전환 시 현재 작업 팀 기준 비교로 교체 필요.
- 칸반: `/api/kanban?team_id=` 파라미터 + `WHERE e.team_id = ?` 필터링 구현됨. 이미 팀 경계 있음.
- 간트: 별도 gantt_task 테이블 없음. events 테이블 데이터를 `project.html`에서 간트로 표시하는 방식. events 팀 경계 완성에 따라 자동 적용됨.

## 8-1. 데이터 소유·편집 권한 모델

팀 자료의 권한 판단 기준을 명시한다. 구현 시 이 규칙에서 벗어나면 안 된다.

**기본 원칙: "팀 공유 자료" 모델**
- 같은 팀의 승인 멤버(`user_teams.join_status = approved` AND `teams.deleted_at IS NULL`)는 그 팀의 자료를 **동등하게** 조회·생성·수정·삭제할 수 있다.
- 자료 row의 `created_by`(작성자 이름)는 표시·로그·이력용일 뿐, **편집·삭제 권한 판단에는 사용하지 않는다**. 즉 같은 팀 멤버라면 누구나 다른 사람이 만든 일정·체크·프로젝트를 수정·삭제할 수 있다. 이건 의도된 동작이며 "개인 소유" 개념이 없다.
- 시스템 관리자(admin)는 전역 슈퍼유저로 모든 팀의 자료를 편집·삭제할 수 있다.

**자료별 적용 표**

| 자료 | 권한 모델 | 비고 |
|------|----------|------|
| 일정(events) | 팀 공유 | `team_id` 일치만 검사. 현재 `auth.can_edit_event`도 동일. |
| 체크(checklists) | 팀 공유 | `team_id` 일치만 검사. 현재 `auth.can_edit_checklist`도 동일. |
| 프로젝트(projects) | 팀 공유 | `team_id` 일치만 검사. 히든 프로젝트는 별도(아래). |
| 칸반·간트 | 팀 공유 | events 데이터 위에 동작하므로 자동 적용. |
| 문서(meetings) | **혼합 모델** | `is_team_doc`으로 분기. 아래 별도. |
| 팀 공지(team_notices) | 팀 공유 | 같은 팀 승인 멤버 누구나 작성·갱신·발송. 운영성 기능이지만 1차 구현은 팀 공유 모델로 둔다. |
| 링크(links, `scope='team'`) | **작성자/admin만 편집·삭제** | 같은 팀 승인 멤버는 조회만 가능. 일정·체크와 다른 예외이며, 링크는 작성자가 직접 큐레이팅하는 자료라는 운영 의도. `scope='personal'`은 항상 작성자 본인 한정. |

**문서(meetings)만의 예외 — 유일하게 "개인/팀" 구분이 있는 자료**

- `is_team_doc = 1` (팀 문서): 팀 공유 모델. 같은 팀 승인 멤버 누구나 편집·삭제 가능.
- `is_team_doc = 0` (개인 문서): **작성자 본인만** 편집·삭제 가능. 다른 멤버는 `team_share = 1`인 경우 읽기만 가능(섹션 8 가시성 규칙). 추방·탈퇴 후에도 작성자 본인은 자기 개인 문서에 계속 접근한다.
- 이 구분은 문서에만 존재한다. 일정·체크·프로젝트에는 "개인" 개념이 없으니 UI도 그렇게 만들어야 한다(예: 일정 작성 폼에 "내 일정/팀 일정" 토글 같은 것을 두지 않는다).

**히든 프로젝트의 추가 예외**

- 같은 팀 멤버라도 `project_members`에 포함되지 않으면 조회조차 불가.
- 편집·멤버 관리·owner 이양은 `projects.owner_id`만 가능. 일반 멤버는 자료 조회·자기 항목 작성만 가능.
- admin은 owner와 동등한 관리 기능을 사용하지만 owner 자리를 차지하지 않는다(섹션 12).

**추방·재가입 시 권한 자동 복구**

- 권한은 row의 동결값이 아니라 매 요청마다 `user_teams` 상태로 판단되므로, 추방되면 자동 차단되고 재가입(`approved` 회복)되면 자동 복구된다. 별도 데이터 마이그레이션·복구 작업이 필요 없다.
- 이는 일정·체크·팀 문서·프로젝트·칸반·간트 모두에 적용된다. 작성자였던 사람이든 아니든 동일하게 처리된다(자기 작성 row여서 보이는 게 아니라 그 팀 소속이라서 보이는 것).
- 개인 문서(`is_team_doc=0`)는 추방과 무관하게 작성자 본인이 항상 보유한다.

## 8-2. 프로젝트 식별자 정책

- 프로젝트명은 팀 안에서만 유일하다.
- 다른 팀에는 같은 이름의 프로젝트가 존재할 수 있다.
  - 예: HW 개발팀의 `A과제`, SW 개발팀의 `A과제`는 별개 프로젝트다.
- 현재 코드의 `projects.name UNIQUE` 전역 유일 제약은 팀 기능 구현 시 제거하고 `(team_id, name_norm)` 유일 정책으로 교체한다.
- 서버/API/DB의 표준 프로젝트 식별자는 `project_id`로 통일한다.
- 일정·체크·문서·간트·칸반은 프로젝트명을 권한 기준으로 직접 쓰지 않고 `project_id` 기준으로 연결한다.
- 화면 표시와 사용자가 입력하는 값은 프로젝트명을 유지하되, 서버 내부 권한 판단과 데이터 연결에는 팀 기준 프로젝트 식별자를 사용한다.
- 히든 프로젝트도 동일하게 팀 소속 프로젝트이며, 같은 이름의 히든/일반 프로젝트 중복 여부는 해당 팀 안에서 판단한다.
- URL이나 API에서 프로젝트를 지정할 때는 가능하면 `project_id`를 사용한다. 이름 기반 구형 API는 전환 기간에만 유지하고, 반드시 `team_id`와 함께 받아 모호성을 제거한다.
- MCP의 `project` 문자열 필터도 `team_id` 없이 단독으로 쓰면 여러 팀의 같은 이름 프로젝트가 섞일 수 있으므로, 장기적으로 `project_id` 또는 `team_id + project` 필터를 제공한다.

## 9. 공개 포털 데이터 노출 정책

비로그인 공개 포털(`/팀이름`)에서 보이는 데이터는 두 단계 조건을 모두 충족해야 한다.

**1단계: 메뉴 노출 여부 (관리자 설정)**
- 시스템 관리자 또는 팀 관리자가 메뉴 관리 페이지에서 팀별로 각 메뉴의 노출 여부를 설정한다.
- **메뉴 노출 설정의 의미는 "공개 포털 UI 진입 차단"이지 "데이터 노출 차단"이 아니다.** 즉 캘린더 메뉴를 X로 두면 `/팀이름`의 캘린더 탭/링크가 사라지지만, 같은 events 데이터가 칸반·간트 메뉴를 통해 노출될 수 있다. 캘린더 형식의 시각화만 외부에 보이지 않게 하는 정책이다.
- 데이터 자체를 외부에 차단하려면 2단계의 항목별 `is_public=0`으로 설정해야 한다.
- 기본값:

| 메뉴 | 기본 노출 |
|------|---------|
| 칸반 | O |
| 간트 | O |
| 문서 | O |
| 체크 | O |
| 캘린더 | X |

**2단계: 항목별 공개 설정 (작성자/프로젝트 설정)**
- 문서·체크: 팀원이 각 항목 작성 시 공개 여부를 직접 선택한다. 공개로 설정된 항목만 포털에 표시된다.
- 캘린더·칸반·간트: 각 이벤트의 `is_public` 값으로 결정된다. 프로젝트 관리 페이지에서 프로젝트 단위로 일괄 설정 가능.
  - 프로젝트 없는 미지정 일정: `is_public` 값으로 결정되지만, 미지정으로 이동 시 서버에서 자동으로 `is_public = 0`으로 강제한다. 사실상 공개 포털에 노출되지 않는다. (이미 구현됨)

**히든 프로젝트 항목 처리**
- 히든 프로젝트(`is_hidden = 1`)에 속한 항목은 `is_public` 설정 값에 무관하게 공개 포털에 표시되지 않는다. 어떠한 설정으로도 외부에 공개할 수 없다.
- 히든 프로젝트 항목은 같은 팀원에게도 노출되지 않는다. `project_members`에 등록된 멤버에게만 표시된다.
- 히든 프로젝트의 이름·내용은 SSE 등 모든 채널에서 차단된다 (섹션 12 동일 원칙).

## 10. 멤버 관리

시스템 관리자와 팀 관리자가 사용하는 멤버 관리 페이지.

기능:
- 팀 선택 드롭박스로 대상 팀 선택
- 신청 멤버 수락 / 거절
- 기존 멤버 추방
- 멤버별 자동 로그인 대상 IP 관리 (**시스템 관리자 전용 — 팀 관리자에게는 UI 자체를 노출하지 않는다**)

거절·추방된 멤버 처리:
- 작성한 데이터(문서, 체크 항목 등)는 유지된다.
- 해당 팀의 `user_teams.join_status`를 `rejected`로 업데이트한다. (row 삭제 아님 — 이력 보존)
- 다른 팀에 여전히 소속된 경우: 기존대로 내부 업무 앱이 유지되며, 해당 팀만 소속 목록에서 사라진다.
- 소속 팀이 하나도 없어진 경우: `/`에서 팀 목록 + 팀 신청 버튼이 표시된다.
- 재신청 시: 해당 `user_teams` row를 `pending`으로 업데이트한다.

권한 범위:
- 시스템 관리자: 모든 팀 조회·수정·관리
- 팀 관리자: `user_teams.team_role = admin`인 팀 목록만 표시·선택 가능
- **팀 관리자의 자기 자신 처리 제한**: 팀 관리자는 자기 자신을 같은 팀에서 추방·강등(`team_role = admin → member`)할 수 없다. 본인 권한 변경은 시스템 관리자만 가능하다. 다른 팀 관리자가 또 있으면 그 관리자가 처리할 수 있다(즉 자기 자신만 막힘).
- **마지막 팀 관리자 보호**: 한 팀에 `team_role = admin`인 멤버가 1명뿐이면 그 사람을 강등·추방·탈퇴 처리할 수 없다. 시스템 관리자가 다른 멤버를 먼저 팀 관리자로 지정한 뒤에만 가능하다.
  - **예외**: 팀이 삭제 예정 상태(`teams.deleted_at IS NOT NULL`)인 경우에는 마지막 팀 관리자 보호를 면제한다. 시스템 관리자가 이미 팀 폐쇄를 결정했고 90일 후 팀이 사라질 예정이므로, 마지막 팀 관리자도 자유롭게 강등·추방·탈퇴 처리할 수 있다. 그 결과 팀 관리자 0명 상태로 잔존하더라도 운영상 문제 없다(시스템 관리자가 멤버 관리 화면에서 직접 처리 가능).

**IP 관리 범위**

`user_ips`는 사용자 전역 설정이다. 팀별 설정이 아니다. whitelist 변경이 모든 팀 접근에 영향을 준다.

권한은 본인·시스템 관리자·팀 관리자 셋으로 나뉜다.

| 작업 | 일반 사용자(본인) | 시스템 관리자(admin) | 팀 관리자 |
|------|:---:|:---:|:---:|
| 본인 현재 PC를 whitelist로 등록 | ✅ 설정 화면 토글 | — (admin 본인은 자동 로그인 대상 아님) | ❌ |
| 본인 whitelist 해제 | ✅ 설정 화면 토글 | — | ❌ |
| 본인 history IP 목록 조회 | ✅ | — | ❌ |
| 임의 사용자의 history·whitelist 조회 | ❌ | ✅ `/admin` | ❌ |
| 임의 사용자에게 새 IP를 whitelist로 직접 등록 | ❌ | ✅ `/admin` | ❌ |
| 임의 사용자의 whitelist 해제·강등 | ❌ | ✅ `/admin` | ❌ |
| 임의 사용자의 history IP를 whitelist로 승격 | ❌ | ✅ `/admin` | ❌ |
| 임의 사용자의 IP row 완전 삭제 | ❌ | ✅ `/admin` | ❌ |

**충돌 처리**
- 같은 IP가 이미 다른 사용자에게 whitelist로 등록돼 있으면 등록을 차단한다. 사용자 자체 등록·admin 직접 등록 모두 동일하게 적용한다.
- admin은 충돌 IP를 다른 사용자에게 이전하려면 기존 whitelist를 먼저 해제·삭제한 뒤 새 사용자에게 등록한다. 한 번에 처리하는 "이전" 단축 기능은 두지 않는다(의도치 않은 변경 방지).

**팀 관리자 정책**
- 팀 관리자에게는 IP 관련 UI·API를 어떤 형태로도 노출하지 않는다(섹션 2).
- 팀 관리자가 멤버 자동 로그인 필요성을 인지하면 운영 채널(메신저·메일 등 시스템 외부 경로)로 시스템 관리자에게 알리고, 시스템 관리자가 `/admin`에서 직접 처리하거나 사용자 본인이 설정 화면에서 등록하도록 안내한다. 시스템 기능에 "요청" 엔드포인트를 두지 않는다.

**다중 소속 사용자 안내**
- 다중 소속 사용자의 IP를 변경하면 모든 팀 접근에 영향을 주므로, 시스템 관리자 UI와 사용자 설정 화면 모두에 이 사실을 명확히 표시한다.

## 11. 팀 관리자 지정

- 팀 생성/관리 페이지에서 시스템 관리자가 해당 팀의 소속 멤버 중 팀 관리자를 지정한다.
- `user_teams.team_role`을 `admin`으로 설정하는 것으로 처리된다.
- 팀 관리자는 여러 명 지정할 수 있다.

## 12. 히든 프로젝트

같은 팀 내에서도 특정 멤버에게만 공개해야 하는 프로젝트를 '히든 프로젝트'로 지정한다. 현재 코드 기준으로는 `projects.is_hidden = 1`, 단일 관리자 `projects.owner_id`, 멤버 테이블 `project_members(project_id, user_id)` 조합으로 이미 구현되어 있다.

권한 규칙:
- 시스템 관리자(admin): 히든 프로젝트 포함 전체 조회·편집 가능. admin은 히든 프로젝트 owner가 아니지만, owner와 동등한 관리 기능(멤버 추가/삭제, owner 지정/변경, 프로젝트 편집)을 사용할 수 있다. 현재 코드도 `is_hidden_project_visible()`에서 admin은 항상 true다.
- 히든 프로젝트 관리자(owner): `projects.owner_id` 1명만 존재한다. owner는 히든 프로젝트 자체 편집, 멤버 추가/삭제, 관리 권한 이양이 가능하다.
- 팀 관리자: 팀 관리자라는 이유만으로 히든 프로젝트를 볼 수 없다. `project_members`에 포함되어야 조회 가능하며, owner가 아니면 멤버 관리·편집은 불가하다.
- 일반 멤버: 히든 프로젝트 멤버에 포함된 경우에만 조회·참여 가능
- 히든 프로젝트의 이름·내용은 SSE 등 공개 채널에도 노출되지 않는다.

**히든 프로젝트 생성 권한:**
- 팀에 소속된 일반 멤버도 히든 프로젝트를 생성할 수 있다.
- **생성 시 `team_id`는 현재 작업 팀 기준으로 결정한다.** 현재 코드 `create_hidden_project`는 owner의 `users.team_id`를 참조하지만, 다중 팀 모델에서는 요청의 `team_id`를 사용하고 `require_work_team_access`로 검증한다.

**히든 프로젝트 관리자 지정:**
- 히든 프로젝트를 생성한 사람이 자동으로 해당 히든 프로젝트의 단일 관리자(owner)가 된다.
- 이미 `projects.owner_id`로 구현되어 있으며, 생성자는 프로젝트 생성 시 자동으로 `project_members`에도 추가된다.
- owner는 기존 멤버 중 한 명에게 관리 권한을 이양할 수 있다. 이양 후 기존 owner는 일반 멤버로 남는다.
- admin은 owner가 아니어도 owner와 동등한 관리 기능을 사용할 수 있다. 현재 코드의 `/api/manage/hidden-projects/{name}/change-owner`처럼 기존 멤버 중 한 명으로 owner를 변경할 수 있고, 팀 기능 구현 후에는 admin도 멤버 추가/삭제 UI를 통해 같은 팀의 승인 멤버를 히든 프로젝트 멤버로 추가할 수 있어야 한다.
- **`owner_id = NULL` 발생 경로**: 코드 `database.py:transfer_hidden_projects_on_removal()`이 처리한다. owner였던 멤버가 팀에서 추방·탈퇴되면 `project_members` 중 같은 팀 활성 멤버 중 가장 먼저 추가된 사람에게 자동으로 owner 이양된다. 후보가 없으면(프로젝트 멤버가 owner 1명뿐이거나, 다른 멤버가 모두 비활성/타팀이면) `owner_id = NULL`로 남는다. 이 row는 admin 외에 누구도 관리할 수 없는 상태가 된다.
- `owner_id = NULL`인 히든 프로젝트가 생긴 경우에도 별도 "원클릭 복구" 기능을 만들지 않는다. admin이 해당 프로젝트의 멤버 관리 화면에서 같은 팀 승인 멤버를 추가한 뒤, 그 멤버를 owner로 지정한다. 즉 admin은 프로젝트 관리자와 동등한 기능을 쓰지만 admin 자신이 owner가 되지는 않는다.
- owner 자신은 멤버에서 바로 제거할 수 없다. 먼저 owner를 다른 멤버에게 이양해야 한다.
- 멤버 추가 후보 조회(`get_hidden_project_addable_members`) 등 기존 헬퍼는 현재 `users.team_id` 단일 비교를 사용한다. 다중 팀 모델에서는 `user_teams.join_status = approved` + `project.team_id` 비교로 교체한다. 특히 `owner_id = NULL` 복구 상황에서는 owner의 팀을 참조하지 말고 반드시 `projects.team_id` 기준으로 후보 조회와 멤버 추가를 처리한다.
- **admin은 멤버 후보 명단에서 자동 제외**한다. admin은 owner도 멤버도 될 수 없으므로(섹션 12 위) 멤버 추가 UI의 후보 드롭다운·검색 결과에 admin을 노출하지 않는다. 같은 원칙은 일정·업무 담당자(`assignee`) 후보, 일반 사용자/멤버 자동완성, MCP 일반 사용자 조회에도 동일하게 적용한다(섹션 2).
- 히든 프로젝트 일정/업무의 담당자(`assignee`)는 현재 코드처럼 프로젝트 멤버만 허용한다.

## 13. 기존 데이터 마이그레이션

팀 기능 도입 전에 작성된 데이터는 팀과 프로젝트 식별자를 함께 보강해야 한다.

**`events.team_id` / `checklists.team_id` NULL 백필 추론 우선순위 (확정)**
1. 같은 row의 `project_id` (또는 백필 후의 `project_id`)가 가리키는 `projects.team_id`를 우선 사용
2. project 매칭이 없으면, 작성자(`created_by` 문자열 → users.name)가 마이그레이션 후 단일 팀(`user_teams.join_status='approved'` row 1건)에 소속이면 그 팀
3. 위 두 단계로도 결정 불가하면 `team_id`를 NULL로 유지하고 `team_migration_warnings`에 row id·매칭 시도 결과 누적. 마이그레이션 자체는 통과시킨다(서버 시작 거부 X). 운영자가 사후 `/admin`에서 정리한다.
- 임의 "기본 팀"으로 일괄 배정하지 않는다(섞이면 분리가 더 어려움).
- 가시성은 NULL row를 보수적으로 처리한다: 작성자 본인 + admin 슈퍼유저만 노출.

**`projects.team_id` NULL 잔존 정책 (events/checklists와 일관)**
- "프로젝트 백필 원칙"의 fallback 단계(아래)로도 결정 불가하면 `projects.team_id`를 NULL로 유지하고 `team_migration_warnings`에 row id 기록. 마이그레이션은 통과시킨다.
- NULL 잔존 projects의 가시성: `owner_id` 본인 + admin 슈퍼유저에게만 노출. 그 프로젝트에 연결된 events/checklists도 같은 제한(가시성 쿼리에 명시).
- 운영자는 사후 `/admin`에서 해당 row를 적절한 팀에 직접 배정한다.

**실행 절차 — 서버 부팅 시 자동 마이그레이션**

운영 워크플로우는 "서버 종료 → 신규 코드로 파일 덮어쓰기 → 서버 시작"이다. 마이그레이션은 별도 CLI 스크립트가 아니라 서버 시작 시 `init_db()`에서 자동 실행한다. 기존 코드의 `_migrate(conn, table, [(col, def), ...])` 헬퍼 + `settings` 테이블의 1회성 마커 패턴(`ev_is_pub_reset_v1` 등)을 그대로 확장한다.

- **자동 백업**: `init_db()` 첫 단계에서 `whatudoin.db` 존재 + 미적용 마이그레이션이 있는 상태이면 `whatudoin.db.bak.{ISO8601_timestamp}` 복사본을 자동 생성한다. 운영자가 별도로 백업할 필요 없다.
- **단계별 마커**: 각 Phase 완료 시 `settings`에 마커를 기록한다(`team_phase1_columns_done`, `team_phase2_backfill_done`, ...). 재시작 시 마커가 있는 단계는 건너뛴다(idempotent 보장).
- **실패 처리**: 마이그레이션 도중 예외 발생 시 트랜잭션 즉시 롤백 + 서버 시작 거부(명확한 에러 로그). 운영자는 자동 백업 파일로 DB를 복구한 뒤 원인을 파악한다. 부분 적용 상태로 서비스가 살아나는 상황을 막는다.
- **동시성**: WhatUdoin은 단일 EXE/트레이 + 단일 SQLite이므로 마이그레이션 동시 실행 위험은 없다.
- **로그 수집**: 마이그레이션 도중의 백필 누락(예: project 문자열 매칭 실패, `team_id` 추론 불가 row)은 stdout 로그 + `settings` 테이블의 별도 키(`team_migration_warnings`)에 JSON으로 누적 기록한다. 운영자가 마이그레이션 후 이 키를 조회해 후속 정리 대상을 확인한다.

**운영자 체크리스트**

운영자가 신규 릴리스를 적용할 때 따라가는 단계는 다음과 같다. 별도 도구·수동 SQL 작업이 필요 없다.

1. **서버 종료**
2. **신규 EXE/코드 파일로 덮어쓰기** (이전과 동일한 워크플로우 그대로)
3. **서버 시작**
4. **시작 로그 확인** — `team_phase1_columns_done` ~ `team_phase4_constraints_done` 마커가 차례로 기록됐는지 본다. 첫 시작에서는 모두 새로 기록되고, 이후 재시작에서는 "이미 적용됨"으로 건너뛴다.
5. **`settings.team_migration_warnings` 조회** — 백필 누락 경고가 있으면 JSON 내용을 보고 후속 정리(수동 팀 배정 등)를 진행한다. 없으면 정상 완료.
6. **자동 백업 파일 보관** — `whatudoin.db.bak.{timestamp}`가 같은 폴더에 생성된다. 운영 안정화 기간 동안 보관하다가 충분히 안전하면 정리한다.

**실패 시 복구 절차**

마이그레이션 실패 시 서버는 시작을 거부하고 stdout/로그에 실패한 Phase·원인을 출력한다. 복구 단계:

1. 서버 종료 상태 확인 (시작 거부됐으므로 이미 정지)
2. 자동 생성된 `whatudoin.db.bak.{timestamp}` 파일을 `whatudoin.db`로 복사해 원래 상태로 복구
3. 로그에서 실패 원인 파악
4. 코드 또는 데이터 수정 후 다시 1~3번 절차 반복

부분 적용된 상태로 운영하지 않는다(트랜잭션 + Phase 마커가 부분 상태를 차단).

**단계별 의존 순서 — Phase 1~5**

다음 순서로 적용한다. 각 Phase는 이전 Phase가 끝났음을 마커로 확인한 뒤에만 진행한다.

| Phase | 내용 | 위험도 |
|-------|------|:---:|
| 1. 컬럼·테이블 추가 + projects 재구성 | `ALTER TABLE`로 컬럼 추가 + 새 테이블 생성 + `projects.name UNIQUE` 제거(테이블 재생성, id 보존). 데이터 값 변경 없음 | 중간 |
| 2. 데이터 백필 | 신규 컬럼에 값 채우기 + hash 변환 후 평문 password 즉시 NULL + admin MCP 토큰 무효화(`mcp_token_hash`/`mcp_token_created_at` 둘 다 NULL) + admin `user_ips` whitelist→history 강등. 트랜잭션 단위로 롤백 가능 | 중간 |
| 3. 시드 정리 | "관리팀" 처리, 초기 시드 제거 | 중간 |
| 4. 제약·인덱스 적용 | UNIQUE 인덱스 + preflight 충돌 검사. 이 단계 실패는 데이터 무결성 문제를 의미 | 높음 |
| 5. 호환 컬럼 drop | `users.team_id`, `users.password`, `events.project`, `checklists.project` 등 제거 | 별도 시점 |

**Phase 1 — 컬럼·테이블 추가 + 프로젝트 테이블 재구성** (`team_phase1_columns_done` 마커)
- `users.name_norm`, `users.password_hash` 컬럼 추가
- `teams.deleted_at`, `teams.name_norm` 컬럼 추가
- `projects.name_norm` 컬럼 추가
- `events.project_id`, `checklists.project_id` 컬럼 추가 (NULL 허용)
- `notifications.team_id` 컬럼 추가
- `team_notices.team_id` 컬럼 추가 (NULL 허용 — 팀별 공지 전환용. Phase 2 백필 + 라우트 전환은 추천 구현 순서의 별도 단계에서 진행)
- 새 테이블 생성: `user_teams`, `team_menu_settings`
- **`projects` 테이블 재구성**: `projects.name UNIQUE` 제거 (Phase 2 백필에서 같은 이름 프로젝트를 다른 팀에 자동 생성해야 하므로 UNIQUE를 먼저 풀어둔다). 테이블 재생성 시 **`projects.id` 보존**: `INSERT INTO new_projects (id, team_id, name, name_norm, color, start_date, end_date, is_active, is_private, is_hidden, owner_id, memo, deleted_at, deleted_by, created_at) SELECT id, team_id, name, NULL, color, start_date, end_date, is_active, is_private, is_hidden, owner_id, memo, deleted_at, deleted_by, created_at FROM projects` 형태로 **명시적 컬럼 목록**으로 복사한다(`SELECT *`는 신규 `name_norm` 컬럼 추가와 컬럼 개수 불일치로 위험). `name_norm`은 일단 NULL로 두고 Phase 2에서 백필. 재생성 후 row count 일치 + `sqlite_sequence` 갱신을 검증한다. 이렇게 해야 `events.project_id`/`checklists.project_id`/`project_members.project_id`/`project_milestones.project_id`/`*.trash_project_id` 매핑을 다시 손대지 않아도 된다. `(team_id, name_norm)` UNIQUE 인덱스 자체는 Phase 4에서 만든다(Phase 2 백필 중 중복 row가 일시 존재 가능). **구현 시 위 컬럼 목록은 실제 `database.py`의 현재 `projects` 스키마와 한 번 더 대조해 누락이 없는지 확인한다(스키마가 갱신되면 마이그레이션 함수 안의 컬럼 목록도 같이 갱신).**
- 기존 컬럼은 그대로 유지(`users.team_id`, `users.password`, `events.project`, `checklists.project` 등)

**Phase 2 — 데이터 백필** (`team_phase2_backfill_done` 마커)
- `users.name_norm` ← `normalize_name(users.name)` (NFC + lower)
- `users.password_hash` ← `hash(users.password)` (admin 포함, 정책 검증 없음)
- **`users.password` 즉시 NULL 처리** (hash 변환 성공 직후 같은 트랜잭션 안에서 평문 컬럼을 NULL로 비운다 — 보안 공백 최소화). 컬럼 자체 drop은 Phase 5에서 일괄 처리.
- `users.team_id` → `user_teams` approved row 이관 (admin 제외, `joined_at = users.created_at`)
- `users.role`: `editor` → `member`
- admin의 `users.team_id` → NULL, `users.mcp_token_hash` → NULL, **`users.mcp_token_created_at` → NULL** (hash와 timestamp를 함께 비워야 UI에 토큰 보유 표시가 잔존하지 않음)
- **admin의 `user_ips` whitelist row → `type='history'`로 강등** (row 삭제하지 않음. 자동 로그인 쿼리는 `type='whitelist'`만 매칭하므로 강등만으로 자동 로그인 효력 제거)
- `team_notices.team_id` 백필 — `created_by` 문자열 → users.name 매칭 → 작성자 단일 팀이면 그 팀, 다중 팀이면 대표 팀(`joined_at` 가장 이른 팀), 작성자 admin/매칭 실패는 NULL 유지 + `team_migration_warnings`에 기록
- `meetings.team_id` 백필 (작성자 admin/team_id NULL 예외 → 별도 처리: 팀 문서는 마이그레이션 로그, 개인 문서는 NULL)
- `events.team_id` / `checklists.team_id` / `projects.team_id` NULL 보강
- `events.project_id` / `checklists.project_id` 백필 (`(team_id, name_norm)` 매칭)
- `notifications.team_id` 백필 (`event_id → events.team_id`, 매핑 불가는 NULL)
- `links.team_id` 백필 (`scope='team'`이고 NULL인 row를 작성자 `users.team_id`로)
- `pending_users` 모든 row 자동 삭제

**Phase 3 — 시드 정리** (`team_phase3_seed_done` 마커)
- admin MCP 토큰 무효화·`user_ips` 강등은 Phase 2에서 처리한다(중복 명시 제거 — 표와 일관).
- 신규 init의 "관리팀" 자동 시드 코드 제거(코드 변경은 이미 반영된 상태로 가정)
- 기존 운영 DB의 `name = '관리팀'` 처리:
  - 참조 데이터 없으면 row 삭제
  - 참조 데이터 있으면 `AdminTeam`으로 rename(`name`·`name_norm` 동시 갱신, 마이그레이션 안에서만 허용되는 1회 예외)

**Phase 4 — 제약·인덱스 적용** (`team_phase4_constraints_done` 마커)
- **인덱스 생성 직전 preflight 충돌 검사**: `users.name_norm`, `teams.name_norm`, `(projects.team_id, projects.name_norm)`, `user_ips` whitelist `ip_address`, `user_teams(user_id, team_id)`, `team_menu_settings(team_id, menu_key)`의 기존 데이터에 중복이 있으면 인덱스 생성 시도 전에 감지하고 서버 시작 거부 + `team_migration_warnings`에 충돌 row 정보 기록한다. 운영자가 자동 백업으로 복구 후 충돌 데이터를 정리하고 재시작한다.
- `CREATE UNIQUE INDEX idx_users_name_norm ON users(name_norm)` (전역, `is_active` 무관)
- `CREATE UNIQUE INDEX idx_teams_name_norm ON teams(name_norm)`
- `CREATE UNIQUE INDEX idx_projects_team_name ON projects(team_id, name_norm)` (Phase 1에서 `projects.name UNIQUE`는 이미 제거됨. 여기서는 새 `(team_id, name_norm)` UNIQUE만 생성)
- `CREATE INDEX idx_events_project_id ON events(project_id)`
- `CREATE INDEX idx_checklists_project_id ON checklists(project_id)`
- `CREATE UNIQUE INDEX idx_user_teams_user_team ON user_teams(user_id, team_id)`
- `CREATE UNIQUE INDEX idx_team_menu_settings ON team_menu_settings(team_id, menu_key)`
- `user_ips`의 `type='whitelist'` `ip_address` 전역 UNIQUE 인덱스 (부분 인덱스)

**Phase 5 — 호환 컬럼 drop** (별도 시점, `team_phase5_drop_done` 마커)
- 이 Phase는 Phase 1~4 적용 후 **충분한 운영 검증 기간**을 거친 뒤 별도 릴리스에서 수행한다(즉시 진행 X).
- 모든 라우트가 `user_teams` / `project_id` 기반으로 전환된 후:
  - `users.team_id` drop
  - `users.password` drop (Phase 2에서 이미 NULL 처리됨. 여기서는 컬럼 자체 제거)
  - `events.project`, `checklists.project` 문자열 컬럼 drop
- SQLite drop은 테이블 재생성이 필요하므로 자동 백업 + 트랜잭션 보호 + 마커로 1회 실행 보장.

**마이그레이션 대상** (아래 항목은 위 Phase 1~5에 매핑된다. 본 항목은 "무엇을 변경해야 하는지"의 평면 목록이며, "언제·어떤 순서로 적용할지"는 위 Phase 단락이 정의한다.)
- `users.team_id` → `user_teams` approved row로 이관 (단, `users.role = admin`인 사용자는 제외 — admin은 `user_teams` row를 만들지 않는다)
- 마이그레이션 후 admin 사용자의 `users.team_id`는 NULL로 비운다.
- `users.team_id` 컬럼 자체는 즉시 drop하지 않는다. 모든 라우트가 현재 작업 팀 파라미터(또는 `user_team_ids`) 기반으로 전환되어 컬럼 참조가 사라진 시점에 별도 마이그레이션으로 drop한다. 호환 단계에서도 신규 쓰기 경로는 컬럼을 갱신하지 않고 `user_teams`만 갱신한다.
- `users.name_norm` 추가. 기존 사용자명의 정규화 값이 충돌하면 마이그레이션을 중단하고 운영자가 먼저 이름을 정리한다. 가입·로그인은 `name_norm` 기준으로 중복/조회한다. `users.name_norm`에는 `is_active` 조건 없는 전역 UNIQUE 인덱스를 둔다.
- **`name_norm` 정규화 규칙**: `users / teams / projects` 모두 동일하게 **Unicode NFC 정규화 + `lower()`** 를 적용한다. 한글 결합형/조합형 차이를 통일하고 ASCII 대소문자를 통일한다. 구현부에는 단일 헬퍼(`normalize_name(s: str) -> str`)를 두고 가입·로그인·중복 검사·인덱스 생성에서 모두 같은 헬퍼를 호출한다. 마이그레이션도 같은 헬퍼로 백필한다.
- **`team_id IS NULL`인 비-admin 사용자**: `user_teams` row를 만들지 않고 그대로 둔다. 마이그레이션 후 그 사용자는 팀 미배정 사용자(섹션 7)와 동일하게 처리되며, 로그인 시 `/`에서 팀 목록 + 팀 신청 버튼이 노출된다.
- `users.role`: `admin`은 유지, 기존 `editor`는 `member`로 전환. 단, `auth.is_editor`/`_require_editor`/`can_edit_*` 등 코드 헬퍼·호출부를 단순 일괄 리네이밍하지 않는다. member 권한과 admin 슈퍼유저 권한을 분리하고, 현재 작업 팀이 필요한 쓰기 작업은 별도 검증 함수를 거친다.
- `users.password` → `users.password_hash`로 전환. 평문 비밀번호는 마이그레이션(Phase 2) 시점에 일괄 hash로 변환하며, 이 변환 단계에서는 비밀번호 정책(섹션 6: 영문+숫자 모두 포함)을 검증하지 않는다. 변환 후에도 사용자는 기존 평문 비밀번호로 그대로 로그인할 수 있다. **hash 변환 성공 직후 같은 트랜잭션 안에서 평문 `users.password` 컬럼을 NULL로 비운다**(보안 공백 최소화). 컬럼 자체는 Phase 5에서 다른 호환 컬럼들과 함께 drop한다. `pending_users` row는 마이그레이션 시 모두 자동 삭제되므로 `pending_users.password` hash 이관 경로는 존재하지 않는다.
- 강제 변경 메커니즘은 도입하지 않는다. `users.password_must_change` 컬럼·관련 미들웨어·외부 IP 차단·IP 자동 로그인 1회 예외는 모두 추가하지 않는다(섹션 6 결정). 마이그레이션 후 정책 위반 비밀번호가 일부 계정에 남을 수 있으나, 운영 인력이 직접 안내해 자발적 변경을 유도한다.
- 비밀번호 정책(영문+숫자) 검증은 계정 가입의 새 비밀번호 입력과 일반 비밀번호 변경 시점에만 적용한다.
- `user_teams` 이관 시 `joined_at`은 해당 사용자의 `users.created_at`을 사용한다. 레거시 데이터는 단일 팀 소속이므로 대표 팀 우선순위 결정에 영향이 없다.
- `teams.deleted_at`, `teams.name_norm` 추가. **`teams` 테이블 재생성은 불필요**하며 `ALTER TABLE`로 컬럼 추가 + `CREATE UNIQUE INDEX ... ON teams(name_norm)`만 적용한다. 기존 `teams.name UNIQUE` 제약은 그대로 둔다(`name_norm` UNIQUE가 더 강한 제약이므로 충돌 없음).
- `pending_users`는 신규 표준 경로에서 사용하지 않는다. **마이그레이션 시 기존 row는 status 무관하게 모두 자동 삭제**한다. 사용자가 신청 대기 중이었다면 새 가입 흐름(`/`의 "계정 가입" 버튼)으로 다시 신청해야 한다. 기존 신청자에게 운영자가 별도로 안내한다. 테이블 자체는 보존하지만 신규 쓰기 경로 없이 빈 상태로 유지한다(향후 별도 마이그레이션에서 drop 가능). `pending_users.team_id`는 추가하지 않는다.
- `events.team_id`, `checklists.team_id`, `meetings.team_id`, `projects.team_id`의 NULL 값 보강
- `meetings.team_id` 백필 규칙
  - `is_team_doc=1`(팀 문서): 기존 `team_id`가 있으면 유지. 없으면 작성자의 기존 `users.team_id`로 백필.
    - **작성자가 admin이거나 작성자의 `users.team_id`가 NULL인 경우**: 자동 백필 대상에서 제외하고 마이그레이션 로그에 별도 기록. 시스템 관리자가 마이그레이션 후 `/admin` 화면에서 해당 row를 직접 조회해 적절한 팀에 수동 배정한다(섹션 13 시드 데이터 정리에서 `AdminTeam`이 살아있다면 1차 임시 배정 대상으로 권장).
  - `is_team_doc=0`(개인 문서): 작성자의 기존 `users.team_id`로 1회 백필. 이후 변하지 않는다.
    - **작성자가 admin이거나 작성자의 `users.team_id`가 NULL인 경우**: 레거시 백필 예외로 `team_id = NULL`로 둔다. 작성자 본인 한정 가시성으로 처리되며 추가 작업 불필요. 단, 팀 기능 구현 후 admin이 새 문서를 작성할 때는 현재 작업 팀이 필요한 쓰기 요청이므로 선택된 `work_team_id` 기준 `team_id`를 저장한다.
- `projects.name UNIQUE` 제거, `projects.team_id`, `projects.name_norm` 기준으로 재구성
- **`events.project_id`, `checklists.project_id` 컬럼 신규 추가** (3단계로 분리 수행):
  1. `ALTER TABLE`로 `project_id INTEGER` 컬럼 추가 (기본값 NULL)
  2. 기존 `events.project` / `checklists.project` 문자열 + 해당 row의 `team_id`로 `(team_id, name_norm)` 매칭해 `project_id` 백필. 매칭 실패 시 별도 로그에 기록.
  3. 인덱스 생성: `idx_events_project_id`, `idx_checklists_project_id`
- `events.project`, `checklists.project` 문자열 컬럼은 호환 전환 단계 동안 읽기용으로 유지하다가, 모든 라우트가 `project_id` 기반으로 전환된 후 별도 마이그레이션에서 drop한다.
- `notifications`에 `team_id` 컬럼 추가. 기존 알림은 `event_id → events.team_id` 매핑으로 우선 백필하고, 매핑 불가하면 NULL로 두고 화면에서 "팀 미상" 라벨을 사용한다.
- **`links` 테이블 정리**: `links.team_id`가 NULL이고 `scope = 'team'`인 row는 생성자(`created_by` 문자열 → users.name 매칭)의 기존 `users.team_id`로 백필한다. `scope = 'personal'`은 `team_id`를 NULL로 유지한다(개인 링크). 매칭 실패한 row는 운영자 검토용으로 로그만 남기고 그대로 둔다. `links.scope`의 의미는 변경하지 않으며, 다중 팀 환경에서 `scope = 'team'`은 그 row의 `team_id` 팀 멤버에게만 노출하는 것으로 해석한다.
- **`team_notices` 팀별 공지 전환**:
  - Phase 1에서 `team_notices.team_id INTEGER` 컬럼 추가 (NULL 허용).
  - Phase 2 백필: `created_by` 문자열 → users.name 매칭 → 작성자가 마이그레이션 후 단일 팀 소속이면 그 팀, 다중 팀이면 대표 팀(`joined_at` 가장 이른 팀), 작성자가 admin이거나 매칭 실패면 NULL 유지 + `team_migration_warnings`에 기록.
  - 공지 조회·저장 라우트(`GET /api/notice`, `POST /api/notice`)는 현재 작업 팀(`work_team_id`) 기준으로 단일 최신 공지 반환·저장하도록 전환. 공지 발송 알림(`POST /api/notice/notify`)도 같은 팀 승인 멤버에게만 전송.
  - 자동 정리(30일/100개)는 팀별로 적용한다. NULL 잔존 row는 운영자가 사후 정리.
  - 팀 완전 삭제 스케줄러(섹션 5)에서 `team_notices`를 `team_id` 기준으로 삭제 대상에 포함.
- `project_milestones.project_id`, `project_members.project_id`, `trash_project_id`는 기존 `project_id` 연결을 유지하되, 프로젝트 테이블 재구성 후 새 id 매핑을 검증
- `/uploads/meetings/...` 기존 파일은 당장 이동하지 않고 읽기 호환을 유지한다. 문서(meetings) 신규 업로드도 기존 `/uploads/meetings/...`를 유지하고, 체크·일정 등 문서 외 신규 업로드부터 `uploads/teams/{team_id}/...`를 사용한다.

**시드 데이터 정리**
- 신규 환경 init에서 "관리팀" 자동 시드를 제거한다. 초기 admin 생성 시점의 `team_id`도 NULL로 둔다.
- 기존 admin 사용자가 있던 `team_id` 값을 NULL로 변경한다.
- 기존 운영 DB에 `name = '관리팀'` 팀이 존재하는 경우의 처리:
  - admin 외에 일반 사용자 소속·프로젝트·일정·문서 등 참조 데이터가 **없으면** 마이그레이션에서 그대로 삭제한다(admin은 위에서 이미 분리됨).
  - 참조 데이터가 **있으면** `AdminTeam`으로 rename(`name`·`name_norm` 동시 갱신)만 하고 그 후에는 **일반 팀과 동일하게** 취급한다. 정렬·UI 분기·MCP 예외 등 별도의 "예약 팀" 개념을 두지 않는다. 시스템 관리자가 이후 멤버 이동·데이터 이전·이름 변경 등을 직접 정리한다.
  - 이 1회 rename은 "팀 이름은 생성 후 변경할 수 없다"(섹션 4) 원칙의 **유일한 예외**이며, 마이그레이션 스크립트 안에서만 수행된다. 일반 운영 화면에서 팀 이름 변경 기능은 제공하지 않는다.

**제거 대상 API/엔드포인트**
- `PUT /api/admin/teams/{team_id}` (현재 `app.py:1614`): 팀 이름 변경 정책 충돌. 마이그레이션에서 **완전 제거**한다. 대응 프론트 UI(팀 이름 수정 입력란)도 함께 제거.
- 팀 이름 수정과 무관한 메타 정보(설명·메모 등)가 향후 필요해지면 별도 컬럼·별도 엔드포인트로 도입한다.

**프로젝트 백필 원칙**
- 기존 `projects.team_id`가 있으면 우선 사용한다.
- `projects.team_id`가 없고 owner가 있으면 owner의 기존 `users.team_id` 또는 이관된 대표 팀으로 배정한다.
- `events.project` 또는 `checklists.project`가 있는데 대응 프로젝트가 없으면, 해당 항목의 `team_id` 안에 프로젝트 row를 생성하고 `project_id`를 연결한다.
- 같은 이름의 프로젝트가 여러 팀에 생길 수 있으므로, 문자열 프로젝트명만으로 전역 update를 수행하지 않는다.
- 모든 백필 후 신규 쓰기 경로는 `project_id`를 저장하고, `project` 문자열은 표시/호환용으로만 유지하거나 단계적으로 제거한다.

**사용자 이름/표시명 주의**
- `users.name`은 로그인 아이디 겸 표시 이름으로 유지한다.
- 현재 DB에 이름 문자열을 저장한 위치(cascade 대상):
  - `events.created_by`, `events.assignee`, `events.deleted_by`
  - `checklists.created_by`, `checklists.deleted_by`
  - `meetings.deleted_by` (작성자는 INTEGER `created_by`로 별개)
  - `projects.deleted_by`
  - `links.created_by`
  - `notifications.user_name`
  - `meeting_locks.user_name`, `checklist_locks.user_name`
  - `checklist_histories.edited_by` (※ `meeting_histories.edited_by`는 INTEGER이므로 cascade 불필요)
  - `team_notices.created_by`
- 이름 변경 기능을 유지하려면 위 컬럼을 모두 cascade 갱신해야 한다.
- 단순성을 우선하면 1차 팀 기능에서는 사용자 이름 변경을 제한하거나, 이름 변경을 시스템 관리자 전용 고위험 작업으로 둔다.

## 14. 첨부파일 저장 정책

저장 경로는 모델별로 다르게 운영한다.

- **문서(meetings) 첨부**: 기존 `/uploads/meetings/YYYY/MM/...` 경로를 그대로 유지한다. 팀별·사용자별 폴더 분리를 적용하지 않는다.
- **체크(checklists)·일정(events) 등 그 외 첨부**: 팀별 폴더 `uploads/teams/{team_id}/...`로 분리한다.

예상 저장 구조:

```text
uploads/
  meetings/
    2026/05/...                 # 문서 첨부 (전역, 팀 구분 없음)
  teams/
    1/
      checks/2026/05/...        # team 1의 체크 첨부
      events/2026/05/...        # team 1의 일정 첨부
    2/
      checks/2026/05/...
```

정리:
- **업로드(쓰기) 시 권한 검증**은 파일 경로가 아니라 **해당 파일을 참조하는 본문(meetings·checklists·events)의 DB 권한**을 기준으로 한다. 폴더 구조는 권한 경계가 아니다.
- 문서 외 첨부 업로드 API는 저장 대상의 `team_id`를 확정한 뒤에만 파일을 저장한다. 사용자가 해당 팀에 접근할 수 없으면 업로드를 거부한다.
- 문서 첨부 업로드는 작성자 권한만 확인한다(문서 자체가 팀/개인/공유 3분류로 권한 분기되므로 폴더 분리는 불필요).
- 팀 이름은 URL 세그먼트로만 노출하고, 디스크 경로에는 사용하지 않는다. 변경 불가능한 `team_id`로 일관성을 유지한다.
- 팀 완전 삭제 시에도 `uploads/teams/{team_id}/...` 폴더는 즉시 삭제하지 않는다. 마찬가지로 `/uploads/meetings/...`도 마이그레이션 등을 이유로 일괄 삭제하지 않는다.
- 별도 고아 파일 정리 작업이 DB 본문·첨부 JSON·휴지통 참조를 스캔해 더 이상 참조되지 않는 파일만 보존 기간 이후 물리 삭제한다. 이 작업은 `/uploads/meetings/...`와 `uploads/teams/{team_id}/...` 양쪽 모두에 적용한다.
- 기존 체크 첨부가 `/uploads/meetings/...`에 함께 저장돼 있던 레거시 파일은 즉시 이동하지 않는다. 신규 업로드부터 위 정책을 적용하고, 기존 파일 물리 이동은 URL 치환·본문/첨부 JSON 갱신·휴지통 참조 검증까지 묶어 별도 마이그레이션으로 분리한다.

**다운로드(읽기) 시 권한 검증**
- 현재 `/uploads/meetings/...`는 단순 정적 공개가 아니다. `app.py`에서 `_ProtectedMeetingStaticFiles`로 마운트되어 있고, 파일 URL이 `meetings.content/attachments` 또는 `checklists.content/attachments`에 참조되어 있으면 해당 문서·체크를 읽을 권한이 있는 사용자에게만 내려준다.
- 즉 목적은 "파일을 업로드해 놓고 문서나 체크를 통하지 않고 링크만 공유해서 권한을 우회하는 것"을 막는 것이다.
- 단, 업로드 직후 아직 문서/체크에 저장되기 전인 파일은 DB 참조가 없으므로 현재 코드는 접근을 허용한다. 에디터 미리보기와 저장 전 첨부 확인을 위해 필요한 동작이다.
- 팀 기능 구현 후에도 `/uploads/meetings/...` 보호 방식은 유지한다. 체크·일정 등 신규 팀별 업로드(`uploads/teams/{team_id}/...`)를 추가하더라도 raw `StaticFiles` 공개가 아니라 같은 권한 검증 래퍼 또는 `/api/files/...` 라우트를 사용한다.
- 인트라넷 운영 전제 때문에 파일 경로를 물리적으로 팀별로 나누는 것은 보조 수단일 뿐이다. 실제 다운로드 권한은 항상 해당 파일을 참조하는 본문(DB row)의 권한으로 판단한다.

## 15. 자동 휴식 페이지 (별도 작업으로 분리)

새벽 2:50 ~ 5:00 동안 접속 시 "자동 정리 중" 안내 페이지를 표시한다.  
새벽 3:00부터 서버 백업·정리 작업이 수행되므로, 2:50부터 서버에 부담을 주지 않도록 사용자를 안내 페이지로 이동시킨다.

> 이 기능은 팀 기능 1차 구현 범위에서 제외한다. 아래 내용은 별도 작업으로 분리할 요구사항 메모이며, 추천 구현 순서에는 포함하지 않는다.

**전환 트리거 메커니즘**
- 모든 내부 업무 페이지에 **공통 가드 스크립트**를 포함한다(예: `static/js/maintenance_guard.js`). 이 스크립트는 클라이언트 시계를 1분 주기로 검사한다.
- SSE 채널에 의존하지 않는다(SSE는 일부 화면에서만 연결되어 있어 전체 커버리지를 보장하기 어렵다). 클라이언트 시계 폴링이 단순하고 페이지 종류를 가리지 않는다.
- 클라이언트 시계 변조 가능성에 대비해 서버 측 게이트도 둔다: 02:50 ~ 05:00에 도착한 모든 일반 API 요청은 서버에서 503/302로 응답한다. 즉 클라이언트 가드를 우회해도 데이터 변경은 차단된다.
- 클라이언트 시계와 서버 시계가 다소 어긋나도 운영 영향은 없다. 02:50 직전에 시작된 요청은 서버 게이트가 막고, 02:30 모달이 02:31에 떠도 사용자는 충분히 대응 가능하다.

**사전 안내 (새벽 2:30)**
- 클라이언트 가드 스크립트가 02:30에 도달하면 모달 창을 띄운다.
  - 내용: "새벽 2:50부터 자동 정리 모드로 전환됩니다."
- 사용자가 모달을 닫고 계속 작업할 수 있다.

**전환 처리 (새벽 2:50)**
- 클라이언트 가드 스크립트가 02:50에 도달하면 현재 페이지 종류를 판별한다.
  - 문서 또는 체크 편집 화면이면 자동 저장 API를 먼저 호출하고, 응답을 받은 뒤 안내 페이지로 이동한다.
  - 그 외 페이지(캘린더, 칸반 등)는 별도 저장 없이 안내 페이지로 이동한다.
- 작성 중인 내용이 유실되지 않도록 저장 완료(또는 명시적 실패 응답)를 기다린 뒤 이동한다.
- **예외: `/admin`은 정리 모드 중에도 그대로 사용 가능하다.** 시스템 관리자의 운영 대응을 위한 예외이며, `/admin` 자체 페이지와 그 하위 admin API(`/api/admin/...`)는 차단하지 않는다. admin이 일반 업무 화면을 슈퍼유저로 사용할 수 있더라도, 정리 모드 중 일반 업무 쓰기 API 허용 범위는 자동 저장·lock 해제 등 최소 allowlist로 제한한다.

**신규 접속 (2:50 ~ 5:00)**
- 해당 시간대에 새로 접속을 시도하면 즉시 안내 페이지가 표시된다.
- 단, 자동 저장, lock 해제/heartbeat, 로그아웃, 정적 자원, 상태 확인처럼 전환 처리를 완료하는 데 필요한 최소 API는 차단하지 않는다. MCP API는 이 예외에 포함되지 않으며 2:50 ~ 5:00 동안 차단된다.
- 문서/체크 자동 저장 API가 성공하거나 명시적으로 실패 응답을 받은 뒤 안내 페이지로 이동한다.

**자동 복귀 (5:00 이후)**
- 5:00이 지나면 안내 페이지에서 자동으로 메인(`/`)으로 이동한다.

## 16. 권한 원칙

- UI가 보낸 `team_id`나 URL의 팀 이름을 그대로 믿지 않는다.
- 서버에서 항상 사용자의 팀 소속 여부를 확인한다.
- 팀 관리자 여부는 `users.role`이 아닌 `user_teams.team_role`로 판단한다.
- 기존의 단일 소속 비교 방식은 다중 소속 모델에 맞게 교체해야 한다.

교체 대상 개념:

```python
item.team_id == user.team_id
```

권장 헬퍼:

```python
is_member(user)                               # role == 'member'. 일반 멤버 여부
is_admin(user)                                # role == 'admin'. 슈퍼유저 여부
user_team_ids(user)                           # approved + deleted_at IS NULL 소속 팀 ID 목록 반환
user_can_access_team(user, team_id)           # approved + deleted_at IS NULL 소속 여부 확인
is_team_admin(user, team_id)                  # user_teams.team_role로 판단
require_work_team_access(user, team_id)       # member는 소속 확인, admin은 비삭제 팀이면 허용
resolve_work_team(request, user, explicit_id) # explicit team_id → work_team_id 쿠키 → 대표 팀 순으로 검증
admin_team_scope(user)                        # users.role=admin이면 전체 팀 범위 반환
```

> 기존 `auth.is_editor`, `_require_editor`, `can_edit_event/checklist/project`는 단순히 `is_member`로 이름만 바꾸면 안 된다. member 경로와 admin 슈퍼유저 경로를 분리하고, 팀이 필요한 쓰기 작업은 `require_work_team_access`를 거쳐 확정된 `team_id`를 사용한다.

> `active_team_id`라는 이름은 사용하지 않는다. 사용자에게 보이는 개념은 **현재 작업 팀**이다.  
> 서버에는 `resolve_work_team` 같은 중앙 헬퍼가 필요하다. 그래야 쿠키, 명시 파라미터, 대표 팀 fallback, 삭제 예정 팀 제외, admin 슈퍼유저 범위를 한 곳에서 일관되게 처리할 수 있다.

의미 구분:
- 현재 작업 팀 기준 화면: 클라이언트가 전달한 `team_id` 사용, `require_work_team_access`로 검증
- 내 스케줄 통합 화면: `team_id IN user_team_ids(user)`
- 외부 공개 화면: URL의 `team_name`으로 팀을 찾고, 공개 설정된 데이터만 반환
- 관리자 화면과 admin 일반 업무 화면: `users.role = admin`이면 전역 팀 범위를 부여한다. 실제 `user_teams` row는 만들지 않는다.

## 16-1. MCP 팀 범위

- MCP read-only 도구의 기본 조회 범위는 현재 사용자가 소속된 모든 승인·비삭제 팀이다.
  - 예: `list_events`, `list_documents`, `list_checklists`, `list_projects`는 기본적으로 내 모든 소속 팀의 열람 가능 데이터를 통합 조회한다.
- 특정 팀만 조회해야 하는 명령을 위해 선택 파라미터로 `team_id` 또는 `team`을 받을 수 있게 한다.
- `team_id` 또는 `team`이 전달되면 서버는 사용자가 해당 팀 소속인지 검증한다.
- **시스템 관리자(admin)는 MCP 토큰 발급 자체가 차단된다.** 설정 화면(`/settings/mcp`)에서 admin 세션은 토큰 생성/재발급 UI가 노출되지 않으며, 토큰 생성 API도 admin 세션이면 403으로 거부한다. admin용 MCP가 필요해지면 별도 admin-only 도구·인증 체계로 분리한다(현 1차에서는 도입하지 않음).
- 마이그레이션 시점에 admin 사용자가 이미 발급해둔 `users.mcp_token_hash`와 `users.mcp_token_created_at`을 **둘 다 NULL로 초기화**하고, 기존 토큰을 무효화한다(hash만 비우고 timestamp가 남으면 UI에서 토큰 보유 표시가 잔존할 수 있음).
- 프로젝트 필터는 `project_id`를 우선 지원한다. 문자열 `project` 필터는 `team_id`가 같이 전달된 경우에만 단일 프로젝트로 해석한다.
- MCP 결과에는 팀 구분이 필요한 항목에 `team_id`와 `team_name`을 포함한다.
- 프로젝트가 있는 결과에는 가능하면 `project_id`와 `project_name`을 함께 포함한다.
- 향후 MCP 쓰기 도구를 추가한다면 팀을 반드시 명시하게 하고, 기본 통합 범위로 쓰기 동작을 수행하지 않는다.
- 시스템 관리자(admin)는 MCP 일반 사용자/멤버 조회 결과에 노출하지 않는다.
- **히든 프로젝트 가시성 — 모든 MCP 도구에 강제 적용**: `list_events` / `list_documents` / `list_checklists` / `list_projects` / `search_*` / `get_*` / `read_*` 등 모든 read-only 도구는 `is_hidden=1` 프로젝트와 그에 속한 events/checklists/meetings를 토큰 사용자가 `project_members`에 포함된 경우에만 결과에 포함한다. 비멤버가 `project_id`로 직접 단건 조회를 시도해도 404로 응답하며, 프로젝트 존재 여부도 노출하지 않는다(이름·내용 모든 채널 차단 — 섹션 12). `team_id` 파라미터로 강제 조회해도 히든 멤버십 검증은 그대로 적용한다. MCP가 히든 프로젝트 우회 경로가 되지 않도록 보장한다.

## 17. 추천 구현 순서

1. DB 마이그레이션 준비: 섹션 13 "실행 절차 + Phase 1~5"를 `init_db()`에 구현한다. 자동 백업, `settings` 마커, 트랜잭션 단위 롤백, 실패 시 서버 시작 거부, 백필 누락 경고 누적 로그까지 포함. Phase 1~4를 본 릴리스에 포함하고, Phase 5(호환 컬럼 drop)는 별도 릴리스로 미룬다. `users.team_id`·`events.project`·`checklists.project`는 이 단계에서 drop하지 않는다.
2. 단일 DB 안에서 `team_id`, `user_teams`(team_role·join_status·joined_at 포함) 모델 정리. 동시에 `users.name_norm`을 추가해 사용자명 대소문자 중복을 차단하고, `users.role`을 `editor → member`로 전환하되, `auth.is_editor`/`_require_editor`/`can_edit_*`는 단순 리네이밍하지 않는다. member 경로와 admin 슈퍼유저 경로를 분리하고, 현재 작업 팀이 필요한 쓰기 작업은 `require_work_team_access`를 거쳐 확정된 `team_id`를 사용한다. `notifications.team_id` 컬럼도 함께 추가한다.
3. 시스템 관리자(admin)를 팀 소속 모델에서 분리하고, 일반 사용자/멤버 목록에서 제외한다. admin은 `/admin` 안에서만이 아니라 일반 업무 화면에서도 슈퍼유저 전역 팀 범위를 가진다. 마이그레이션에서 기존 admin의 `users.team_id`를 NULL로 비우고, 신규 init 시드의 "관리팀" 자동 생성을 제거한다.
4. 기존 문서·체크·일정·프로젝트 데이터를 특정 팀 또는 추론 팀에 배정해 `team_id`를 먼저 확정
5. 프로젝트 식별자 정리: `project_id` 표준화, 팀별 프로젝트명 중복 허용, `projects.name UNIQUE` 제거, `(team_id, name_norm)` 유일 정책 적용
6. 기존 events/checklists/projects/milestones/project_members/trash 연결을 `project_id` 기준으로 백필
7. **로그인 인증 기반 정비 (#8과 한 페이즈로 함께 진행 — #7 → #8 의존)**: 일반 사용자 로그인은 비밀번호 단독 → 이름(아이디)+비밀번호 입력으로 **전환**(폐기 후 신규 도입), `users.password_hash` 컬럼·평문→hash 일괄 변환. `users.name_norm`은 #2에서 이미 추가됐으므로 이 단계에서는 다시 추가하지 않고 로그인·중복 검사 경로만 갱신한다. admin 로그인은 현재처럼 별도 `/api/admin/login` + 짧은 세션을 유지. **일반 `/api/login`은 `users.role = admin` 사용자를 조회 대상에서 제외**해 admin 이름 노출을 차단한다. 강제 변경 메커니즘·외부 IP 차단·IP 자동 로그인 1회 예외는 도입하지 않는다(섹션 6 결정).
8. 계정 가입과 팀 신청 분리: 계정 가입은 이름/비밀번호만 받고 즉시 팀 미배정 `users` row를 생성한다(예약 사용자명 차단, 사용자명 대소문자 중복 차단, 비밀번호 중복 검사 폐기 포함, `users.role=member`, `team_id=NULL`, `name_norm` 정규화 저장). 가입 완료 직후 자동 세션 생성 + `/`로 리다이렉트. 팀 신청은 로그인 후 `user_teams.join_status`로 관리하며, 거절/추방 후 재신청은 같은 `(user_id, team_id)` row를 `pending`으로 갱신한다. **비밀번호 정책(영문+숫자) 검증은 계정 가입과 일반 비밀번호 변경 시점에만 적용**한다.

## 18. 현재 결론

- 지금은 DB를 여러 개로 쪼개지 않는다.
- 먼저 단일 DB에서 팀 경계를 제대로 잡는다.
- 팀 관리자 여부는 `user_teams.team_role`로만 판단하며, `users.role`은 `admin / member` 두 가지만 사용한다.
- 프로젝트명은 팀별로 중복될 수 있으므로, 구현 시 프로젝트 식별 기준을 먼저 정리한다.
- 시스템 관리자(admin)는 실제 팀 소속 없이 웹앱 슈퍼유저 전역 팀 범위를 가진다. `/admin`은 운영 기능 중심으로 유지하고, 일반 자료 편집은 기존 업무 화면에서 수행한다. 단, MCP 토큰 발급은 1차 구현에서 차단한다.
- 삭제 예정 팀은 대표 팀/현재 작업 팀 후보에서 제외하며, 삭제 예정 팀만 남은 사용자는 팀 미배정 사용자처럼 처리한다. 팀 미배정 사용자에게도 본인 작성 개인 문서 통합 노출용 "내 자료" 영역을 제공하되, 알림은 노출하지 않는다.
- 활성 팀이라는 표현 대신 **현재 작업 팀**을 사용한다. 식별자는 쿠키(`work_team_id`)를 1차 초기값으로 사용하며, localStorage는 백업/UX 보조 용도다.
- 팀 이름은 생성 후 변경 불가이며, 운영 화면에서는 변경 UI/API를 제공하지 않는다(마이그레이션의 1회 rename은 예외).
- 계정 가입과 일반 비밀번호 변경에서 정책(영문+숫자) 검증을 적용한다. 마이그레이션 시 평문 → hash 일괄 변환만 정책을 검증하지 않는다.
- 사용자명은 대소문자 무관으로 중복 차단한다. `Kim`과 `kim`은 같은 계정이다. 일반 사용자 로그인은 이름+비밀번호로 전환하고, admin은 별도 `/api/admin/login` + 짧은 세션을 유지한다.
- 비활성 사용자도 사용자명을 계속 점유하므로 `users.name_norm`은 `is_active`와 무관하게 전역 UNIQUE로 유지한다.
- IP 자동 로그인은 두 경로로 등록한다 — 일반 사용자가 설정 화면에서 본인 PC를 직접 등록하거나, 시스템 관리자가 `/admin`에서 임의 사용자에게 임의 IP를 부여한다. 팀 관리자에게는 IP 관련 UI·API를 노출하지 않는다.
- 팀 기능 도입 이후 문서(meetings) 신규 업로드는 기존 `/uploads/meetings/...`를 유지하고, 체크·일정 등 문서 외 신규 업로드는 팀별 폴더에 저장한다. 기존 전역 업로드 파일은 읽기 호환을 유지하고, 물리 이동/삭제는 별도 마이그레이션과 고아 파일 정리 작업으로 분리한다.
- 첨부파일은 인트라넷 운영 전제에서도 현재 `/uploads/meetings`처럼 DB 참조 기반 권한 검증을 유지한다. 링크만 공유해 문서·체크 권한을 우회하는 것은 막는다.
- 자동 휴식 모드는 팀 기능 1차 구현 범위에서 제외하고 별도 작업으로 분리한다.
- MCP 기본 조회 범위는 사용자의 모든 소속 팀이며, 필요 시 팀 필터를 명시한다.
- 팀별 DB 또는 `common.db + team DB`는 나중에 실제 단일 DB 한계가 확인될 때 검토한다.
- 마이그레이션은 별도 CLI 없이 서버 부팅 시 `init_db()`에서 자동 실행한다. 자동 백업 + `settings` 마커 + 단계별 트랜잭션 + 실패 시 서버 시작 거부의 4중 안전장치를 둔다(섹션 13 실행 절차).
- 이 문서는 현재까지의 설계 논의 정리이며, 구현 시점에는 실제 코드와 다시 대조해 세부 계획을 확정한다.
