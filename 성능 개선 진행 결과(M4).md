## M4-0 완료 (2026-05-10) — M4 진입 게이트 평가

- **결정**: **target-risk override 진입**(M2-0/M3-0과 동일 패턴). 정량적으로는 NO-GO 분기지만 외부 의존 격리 가치를 인정해 진입한다.
- **NO-GO 분기 근거**:
  - M1c-ULTRA(Ollama limiter 7접점 + admin UI 1~5 + busy/unavailable 통합 UX) 적용 후 외부 Ollama hang/ConnectionError/메모리 누수가 main app 응답성에 측정 가능한 회귀를 만든다는 기록 없음.
  - §13 진입 게이트 조건 "5분 timeout 내 정상 처리되는 hang, ConnectionError, 메모리 누수가 main app에 미치는 영향이 측정 가능한 수준으로 남을 때만 진입" 미충족.
  - §16 "M1 단계 ①(세마포어)으로 충분한지 부하 테스트와 운영 관찰로 판단한다. hang/메모리 누수가 관찰될 때만 진행" 조건 정량 미충족.
- **GO override 근거**:
  1. M2 supervisor 골격(M2-9/17) + M3 scheduler service 분리 패턴(M3-2/3) 안정화. 추가 sidecar 분리 비용 매우 낮음 — `WhatUdoinSupervisor`/`ServiceSpec`/`scheduler_service_spec` factory 패턴/`probe_healthz`/`STOP_ORDER` 재사용 가능.
  2. 외부 Ollama 서버 자체가 메모리 누수/hang 위험을 가진 외부 의존. 사내 LAN에서 Ollama 모델 교체/재시작/장애가 운영 중 발생할 가능성 — main app threadpool 잠식 위험을 사전 격리.
  3. M1c-9 통합 UX(busy/unavailable)가 Ollama service 분리 후에도 동일 동작해야 함. 분리 시 회귀 방지를 위해 코드 경로를 명시적으로 정리하는 가치.
  4. M4 종료 후 §12 uvicorn `limit_concurrency` 결정이 가능해진다(분기 (가): sidecar 도입 후 main app threadpool 부하 모델이 단순해짐).
  5. 사용자 명시 진입 요청.
- **target-risk 표**:

  | 리스크 | 정량 측정 | override 사유 |
  |---|---|---|
  | Ollama hang으로 main app threadpool 잠식 | 미관측(M1c-ULTRA limiter 7접점 + 5분 timeout) | 외부 의존 격리 가치 + supervisor 패턴 재사용 비용 낮음 |
  | Ollama 메모리 누수가 main app으로 전파 | 미관측 | 프로세스 격리로 원천 차단 |
  | M1c-9 통합 UX 회귀 | 미관측 | 분리 후에도 동일 동작 보장 — 회귀 방지 단언 가치 |
- **변경 파일**: 없음. 본 step은 결정 step.
- **다음 step 영향**: M4-1에서 `/internal/llm` JSON IPC 채널 + `ollama_service.py` 신설. M2-16 publisher.py + M3-2 scheduler_service.py 패턴 재사용. M4-2에서 M1c-9 통합 UX 코드 경로(현재 `app.py`에 분산)가 service 분리 후에도 동일 동작하는지 검증. M4-3은 분리 완료 후 §12 uvicorn `limit_concurrency` 결정.

## M4-3 완료 (2026-05-10) — uvicorn limit_concurrency 미적용 채택

- **결정**: 마스터 plan §12 sidecar 도입 후 (가) 분기 후보 중 **`limit_concurrency` 미적용 채택**. SSE service는 §12 명시적 미적용. 운영 코드 변경 0건(현재 모든 uvicorn.Config/run 호출에 `limit_concurrency`/`limit_max_requests` 인자가 부재).
- **사유**:
  1. M2~M4 sidecar 분리 완료 후 main app uvicorn은 SSE 장기 연결을 보유하지 않음(SSE service 분리). LLM 호출은 main app에서 IPC 한 hop으로 끝남(Ollama service 분리). main app threadpool 잠식 시나리오 0.
  2. 사내 운영 환경(n≈1~5명) 폭주 시나리오 미관측. M3-4 라이브 측정 일반 API p95 12.9ms(SLA 500ms 대비 38배 여유). 503 보호가 필요한 한계 폭주 미관측.
  3. 보호는 §8 Ollama 세마포어(M1c-ULTRA + M4-1 service 분리) + §7 업로드 세마포어(M1c-ULTRA) + WAL/busy_timeout(M1b-ULTRA)으로 충분.
  4. SSE service `limit_concurrency` 적용 시 SSE 503 차단으로 알림 정합성 깨짐(plan §12 SSE service 보호 절).
  5. `limit_max_requests`는 단일 EXE 운영에서 worker 재시작 문제(plan §12 금지 항목). 미적용 유지.
- **운영 코드 검증**: `uvicorn.Config`/`uvicorn.run`/`uvicorn.Server` 호출 10건(main.py HTTP/HTTPS + app.py 직접 launcher + sse_service.py + scheduler_service.py + ollama_service.py)에 `limit_concurrency` 0건 grep 단언 PASS, `limit_max_requests` 0건.
- **회귀 테스트 추가**: `tests/phase68_uvicorn_limit_concurrency_not_applied.py`. 운영 5개 파일에서 `limit_concurrency`/`limit_max_requests` 키워드 0건 grep + uvicorn Config/run/Server 호출 ≥ 4건 단언. 향후 누군가 이 두 키워드를 추가하면 즉시 회귀 검출.
- **증거**: phase68 3/3 PASS. uvicorn 호출 10건 모두 검사됨.
- **명령**: `D:\Program Files\Python\Python312\python.exe tests\phase68_uvicorn_limit_concurrency_not_applied.py` PASS. phase54~67 회귀 모두 PASS.
- **재평가 트리거**: 다음 시나리오 관측 시 본 결정 재평가:
  - 동시 사용자 수가 실제 50+ 수준으로 증가
  - 일반 API p95가 SLA 500ms 근처로 회귀
  - sidecar 분리 후에도 main app threadpool 잠식이 측정으로 확인
  위 셋 중 하나라도 발생하면 (가) 분기 후보의 200 이상 보수적 적용 또는 (나) 분기 후보 검토.
- **다음 step 영향 (M4-4)**: M4-2의 hang 시뮬레이션 한계(uvicorn threadpool 격리 라이브 회귀)를 M4-4 종료 부하에서 보강. 라이브 supervisor 통합(Web API + SSE + Scheduler + Ollama 4 service 동시 spawn) → Ollama hang 중 main app `/api/health` 50회 동시 GET p95 측정.
