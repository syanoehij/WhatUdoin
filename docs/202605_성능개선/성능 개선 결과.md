## 1. 마일스톤별 성과

### M1a — 프론트엔드 lazy-load + baseline 측정

- 홈/체크/프로젝트 관리/휴지통 viewer 보조 화면에서 Tiptap/Mermaid/WUEditor 대형 번들 사전 로드 제거 → 첫 렌더링 전에 다운로드/parse/eval 발생 0건
- locust 1/5/10/25/50 VU 단계별 baseline 측정 인프라 구축
- §5-1 4단계 측정(번들 다운로드 / parse·eval / `WUEditor.create()` / viewer 표시) before/after 기록
- viewer 기능 회귀 0건

### M1b-ULTRA — SQLite WAL + PRAGMA

- `database.get_conn()`에 PRAGMA 5종 적용: `timeout=5` / `busy_timeout=5000` / `synchronous=NORMAL` / `cache_size=-8000` / `temp_store=MEMORY`
- `PRAGMA journal_mode=WAL` 1회 활성화
- env override 가능 — `WHATUDOIN_SYNCHRONOUS_MODE=FULL` 시 보수 내구성
- 50 VU 부하에서 `database is locked` **0건** 유지 (재측정 p95 3000ms, baseline 5300ms 대비 개선)

### M1c-ULTRA — Ollama limiter + admin UI + 통합 UX

- 외부 Ollama HTTP 호출 7접점 모두 limiter로 감쌈(파싱/refinement/체크리스트 생성/주간 보고/conflict review/health/model)
- admin UI에서 1~5 슬롯 설정 + DB persist + live capacity 변경
- busy/timeout/connect/5xx 모두 `OllamaUnavailableError` → "AI 사용 중 (N/N)" / "AI 사용 불가" 통합 UX

### M1d — MCP 병목 측정 (skip 종료)

- MCP 검색 중 웹 UI/SSE 정지 또는 일반 API p95 회귀 미관측
- 게이트 미통과로 마일스톤 종료. transport 교체/threadpool 분리 전부 미수행

### M2 — SSE service 분리 + Front Router + 보안 11종

- SSE broker를 `sse_service.py`로 분리(Starlette + 자체 SSEBroker + `/api/stream` + `/internal/publish` loopback guard)
- `publisher.py` publish 추상화 — env 분기로 IPC vs in-process fallback
- `front_router.py` ASGI dispatcher + path별 라우팅(`/api/stream`/`/uploads/meetings/*`/`/internal/*` 차단)
- `supervisor.py` 골격 — service spec factory + 토큰 발급 + crash-loop + STOP_ORDER + STARTUP_SEQUENCE
- **보안/정책 11종**: 외부 포트 owner / SSE 접근 경로 / canonical URL / HTTP fallback 정책 / HSTS 결정 / AVR 호환 / strip-then-set forwarded headers / SlowAPI limiter trusted-proxy / TRUSTED_PROXY 외부 직접 차단 / CSRF Host 보존 / HTTPS probe 호환
- 정책별 증거 인덱스 18종 + 라이브 supervisor 통합 14/14 + 50 SSE 부하 10/10 PASS

### M3 — Scheduler service 분리

- `scheduler_service.py` 신설 — APScheduler + cron 6종 + finalize_expired_done startup 콜백 + healthz 4종 키 + RotatingFileHandler 10MB×14

- supervisor STOP_ORDER에 scheduler 추가, M2_STARTUP_SEQUENCE 8→9→10 항목으로 진화
- 라이브 supervisor 통합 14/14 + owner 정책 위반 3종 0건 21/21 + 일반 API p95 **12.9ms** PASS

### M4 — Ollama service 분리 + uvicorn 결정

- `ollama_service.py` 신설 — Starlette + `/internal/llm` token Bearer + loopback guard + busy/timeout/connect/5xx 응답 분기

### M5 — Media service 분리

- `media_service.py` 신설 — Starlette + `/internal/process` token Bearer + loopback guard + staging path 정규화 + PIL.verify 이중 open + ext whitelist + dimensions + SHA-256

- 20MB/10MB 혼합 업로드 thread 진행 중 일반 API p95 **78ms** (6× 여유)

### M6 — MCP write owner boundary

- mcp_server SQL write 패턴 0 / sqlite3 직접 import 0 / db.write 호출 0 (AST + grep 검사)

### 사용자 의도 회복 fix 4종 (가장 중요)

토요일 진행 도중 plan §13 권장안이 사용자 운영 모델과 충돌했고, 다음 4건을 다시 정리:

1. **MCP HTTP 화이트리스트 추가** → M2-4 가드 제거로 자연 흡수
2. **`/settings/mcp` 안내 URL scheme 분기** — HTTP 접속자에게 HTTP URL 안내, HTTPS 접속자에게 HTTPS URL 안내
3. **M2-4 정책 재결정 — HTTP unsafe write 가드 제거** — 사내 인트라넷 망 전제에서 HTTP/HTTPS 기능 동등, HTTPS는 알람(Notification API secure context) 전용
4. **session cookie secure flag 동적 분기** — `secure=(request.url.scheme == "https")`로 HTTP 접속에서도 cookie 저장 가능 → HTTP 로그인 회복

## 4. 분리 활성화 가이드 (env 토글)

### 일상 운영 (default, fallback)

```powershell
python main.py
```
→ 단일 프로세스. 토요일 이전 동작과 100% 동일. SQLite WAL/Ollama limiter/HTTP 로그인 fix 등은 자동 적용.

### 단계별 sidecar 활성화

```powershell
# 1단계 — Scheduler 격리 (cron이 main app 재시작에 영향 안 받음)
$env:WHATUDOIN_ENABLE_SCHEDULER_SIDECAR="1"

# 2단계 — Media 격리 (PIL/디스크 I/O 격리)
$env:WHATUDOIN_ENABLE_MEDIA_SIDECAR="1"

# 3단계 — Ollama 격리 (LLM hang/메모리 누수 격리)
$env:WHATUDOIN_ENABLE_OLLAMA_SIDECAR="1"

# 4단계 — Front Router + Web API internal-only + SSE 분리
$env:WHATUDOIN_ENABLE_FRONTEND_ROUTING="1"

python main.py
```

### 모든 분리 활성화 시

6 service 동시 spawn:
- Front Router (8000/8443 외부 owner)
- Web API (127.0.0.1:8769 internal-only)
- SSE service (127.0.0.1:8765)
- Scheduler service (127.0.0.1:8766)
- Ollama service (127.0.0.1:8767)
- Media service (127.0.0.1:8768)

종료 시 STOP_ORDER `(ollama, media, sse, scheduler, web-api, front-router)` 순서로 graceful stop.

### 복구

degraded 시 토글을 끄고 `python main.py` 재시작 → fallback 즉시 복귀.

## 8. 결론

토요일 오전 7시 이후 33시간 동안 진행한 사이클 결과:

- **사용자 경험은 그대로** — UI/UX/메뉴/로그인/일상 동작 변경 없음. HTTP/HTTPS 기능 동등 + 알람만 HTTPS 전용이라는 원래 운영 모델 100% 보존.
- **외부 의존(Ollama/PIL/디스크 I/O/cron) 격리 가능** — env 토글로 단계별 활성화. 일상은 fallback, 운영 회귀 시점에 즉시 분리.
- **회사 반입을 위한 인프라 준비 완료** — supervisor + 4 service + Front Router + 26 phase 회귀.
- **모든 측정값 SLA 대비 큰 여유** — 일반 API p95 12.9~78ms (SLA 500ms 대비 6~38×).
- **사용자 의도 회복 4건** + **정책 정비 11종** — 회선 신뢰는 §2 사내 LAN 전제 + M2-11 strip-then-set + M2-13 외부 차단으로 보호.

다음 분기점은 **회사 반입 결정** 또는 **MCP write tool 운영 요구 발생** 시. 현재 코드 boundary가 후속 작업의 비용을 zero-rework에 가깝게 낮춰둔 상태다.
